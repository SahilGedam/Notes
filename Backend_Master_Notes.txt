1. CLIENT-SERVER ARCHITECTURE & COMMUNICATIONS [Importance: 8.5/10]
--------------------------------------------------------------------------------
* Monolith vs Microservices Architecture:
  - Monolith: Single, unified codebase and deployment unit. Easy to develop, test,
    and deploy initially, but harder to scale, maintain, and adopt new technologies
    as the application grows.
  - Microservices: Application split into small, autonomous, loosely coupled services.
    - Each service manages its own database and communicates over lightweight protocols
      (HTTP/REST, gRPC, message queues).
    - Pros: Independent scaling, technology diversity, clear team boundaries.
    - Cons: Network latency, complex distributed data management (Saga pattern),
      operational overhead, and distributed debugging.

* API Architectures:
  - REST Principles (Representational State Transfer):
    - Resource-oriented API architectural style driven by 6 core constraints:
      1. Client-Server: Separation of UI/concerns from database/storage concern.
      2. Stateless: Every HTTP request must contain all context needed to process it. No session stored on server.
      3. Cacheable: Responses must state if they can be cached to prevent client resource reloading.
      4. Uniform Interface: Standard URIs, resource representation manipulations (e.g. JSON), self-descriptive messages, and HATEOAS.
      5. Layered System: Clients interact with gateways, proxies, or load balancers transparently.
      6. Code on Demand (Optional): Sending executable scripts (like JS) to client dynamically.
    - REST Resource Naming & Nested Resources:
      - Use plural nouns for resource endpoints: `/users`, `/products`, `/orders`.
      - Represent hierarchical relationships using nested resources: `/users/5/orders` (gets orders belonging to user 5).
      - Maintain clean resource naming: avoid verbs in URIs (e.g., use `POST /users` instead of `POST /createUser`).
    - CRUD Mapping: Mapping DB CRUD operations to standard HTTP verbs:
      - Create -> `POST /resources`
      - Read -> `GET /resources` or `GET /resources/:id`
      - Update (Replace) -> `PUT /resources/:id` (idempotent, replaces entire object)
      - Update (Modify) -> `PATCH /resources/:id` (non-idempotent/idempotent, modifies specific fields)
      - Delete -> `DELETE /resources/:id` (idempotent)
    - HATEOAS (Hypermedia As The Engine Of Application State):
      - A REST principle where responses return hypermedia links to guide the client on valid operations they can perform next.
      - Example JSON:
        {
          "id": 42,
          "name": "Widget",
          "price": 19.99,
          "_links": {
            "self": { "href": "/products/42" },
            "purchase": { "href": "/products/42/checkout", "method": "POST" }
          }
        }
    - Suffers from over-fetching (retrieving unnecessary fields) or under-fetching
      (requiring multiple endpoints/requests for a single view).
  - GraphQL:
    - Schema-driven, single-endpoint query language.
    - Clients specify exact fields they need, eliminating over/under-fetching.
    - N+1 Query Problem: Occurs when a resolver triggers database queries in a loop
      for nested relations. Solved using batching/caching libraries (e.g., DataLoader).
  - gRPC:
    - High-performance RPC framework developed by Google.
    - Uses Protocol Buffers (protobuf) for binary serialization (smaller/faster payloads).
    - Runs on HTTP/2, allowing multiplexed streams, bidirectional streaming, and header
      compression.
    - Best for internal microservice-to-microservice communication.
  - WebSockets:
    - TCP-based protocol establishing a persistent, bidirectional, full-duplex connection
      after an initial HTTP/1.1 handshake.
    - Best for real-time applications (chat, live notifications, real-time gaming).
  - Webhooks:
    - Reverse APIs or HTTP push notifications.
    - A server sends HTTP POST requests containing event payloads to registered client URLs
      when specific events occur. Fully asynchronous and event-driven.


2. HTTP PROTOCOL DEEP DIVE [Importance: 9/10]
--------------------------------------------------------------------------------
* HTTP Methods, Safety, and Idempotency:
  - Safe Method: Does not modify server state (read-only).
  - Idempotent Method: Making multiple identical requests has the same server state
    as making a single request.
  
  -----------------------------------------------------------------------------
  Method   | Description                     | Safe? | Idempotent?
  -----------------------------------------------------------------------------
  GET      | Retrieve resource representation| Yes   | Yes
  POST     | Create a new resource           | No    | No
  PUT      | Replace/Upsert resource         | No    | Yes
  PATCH    | Partially modify resource       | No    | No (Often implemented as yes)
  DELETE   | Remove resource                 | No    | Yes
  HEAD     | Same as GET but no body returned| Yes   | Yes
  OPTIONS  | Describe communication options  | Yes   | Yes
  -----------------------------------------------------------------------------

* HTTP Status Codes:
  - 1xx (Informational): e.g., 101 Switching Protocols (WebSockets).
  - 2xx (Success):
    - 200 OK: Request succeeded.
    - 201 Created: Resource successfully created.
    - 204 No Content: Request succeeded, but response has no body.
  - 3xx (Redirection):
    - 301 Moved Permanently: Resource moved to a new URI permanently.
    - 302 Found / 307 Temporary Redirect: Temporary redirect; original method preserved in 307.
    - 304 Not Modified: Resource has not changed (uses cache validation headers).
  - 4xx (Client Errors):
    - 400 Bad Request: Malformed request syntax or invalid input validation.
    - 401 Unauthorized: Lacks valid authentication credentials (identity unknown).
    - 403 Forbidden: Identity known, but user lacks permissions (authorization failed).
    - 404 Not Found: Requested resource could not be found.
    - 409 Conflict: Request conflicts with current server state (e.g., duplicate unique keys).
    - 429 Too Many Requests: Rate limit exceeded.
  - 5xx (Server Errors):
    - 500 Internal Server Error: Generic catch-all server crash.
    - 502 Bad Gateway: Proxy/Gateway received an invalid response from upstream server.
    - 503 Service Unavailable: Server overloaded or down for maintenance.
    - 504 Gateway Timeout: Upstream server failed to respond in time.

* Key HTTP Headers:
  - `Authorization`: Contains credentials (e.g., `Bearer <token>`).
  - `Cookie` / `Set-Cookie`: Manages client-side session identification state.
  - `Content-Type`: Format of the request/response payload (e.g., `application/json`).
  - `Cache-Control`: Caching directives (e.g., `no-cache`, `max-age=3600`).
  - CORS Headers: `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, etc.

* Server-side validation vs Client-side validation:
  - Client-side validation (HTML/JS) runs in the browser, providing instant feedback but is easily bypassed by intercepting requests.
  - Server-side validation is the final mandatory defense line to validate data schemas, required fields, and sanitize inputs before persistence.

* HTTP Request Lifecycle (Detailed Step-by-Step):
  1. URL Parsing & DNS Resolution: Browser parses scheme, host, port, path. It checks caches (browser, OS, router, ISP) for the host's IP. If not found, recursive DNS query: Root Domain (.) -> TLD (.com) -> Authoritative Nameserver -> returns IP.
  2. TCP 3-Way Handshake: Client and server establish connection via Layer 4 TCP protocol: SYN -> SYN-ACK -> ACK.
  3. TLS/SSL Handshake (Secure HTTPS): Client and server negotiate cipher suites. Server presents certificate (checked against browser trust store). A symmetric session key is generated via Diffie-Hellman/ECDH for encrypting the session.
  4. HTTP Request Transmission: Client sends the request packet (Request Line, Headers, and Body) over the encrypted TLS socket.
  5. Load Balancing & Reverse Proxy: Traffic hits the gateway (Nginx, ALB). The reverse proxy terminates TLS, handles caching, and forwards the raw request to the target backend application instance.
  6. Application Processing: Web application routing framework (e.g. Express or Spring Boot) passes request through middlewares (CORS, Rate Limit, Auth verification, Body parsers, validation) to Controller -> Service -> Repository -> Database Query.
  7. HTTP Response Generation: Controller formats data into a DTO and serializes it to JSON. Server writes status code, headers (Content-Type: application/json), and the body string to socket.
  8. Connection Reuse or Teardown: If header `Connection: keep-alive` (default in HTTP/1.1 and 2), TCP socket remains open to process next requests. Otherwise, TCP teardown occurs (FIN -> ACK -> FIN -> ACK).

* URI vs URL:
  - URI (Universal Resource Identifier): Unique identifier for a resource (name, locator, or both).
  - URL (Universal Resource Locator): A subset of URI that specifies the resource identifier AND the access protocol/mechanism to locate it (e.g., `https://example.com/index.html`).
  - URN (Universal Resource Name): A subset of URI that identifies a resource by a persistent name in a namespace, without location info (e.g., `urn:isbn:0451524934`).
  - Principle: All URLs are URIs, but not all URIs are URLs.


3. AUTHENTICATION, AUTHORIZATION & SECURITY [Importance: 9.5/10]
--------------------------------------------------------------------------------
* Session-Based (Stateful) vs. JWT-Based (Stateless) Authentication:
  - Stateful: Server creates a session ID, stores it in memory/database (Redis),
    and sends it in a cookie. The client sends the session ID back with each request.
    - Pros: Instant revocation of sessions is easy.
    - Cons: High server memory footprint, scaling requires shared session stores.
  - Stateless: Server signs a JSON Web Token (JWT) and returns it. Client stores it
    (localStorage or HTTP-only cookie) and attaches it in the Authorization header.
    - Pros: Highly scalable, no session store needed, good for multi-domain APIs.
    - Cons: Hard to revoke tokens before expiration (requires token blacklisting).

* OAuth 2.0 & OpenID Connect (OIDC):
  - OAuth 2.0: Delegation framework for authorization. Permits third-party apps
    to access resources without sharing passwords.
    - Key grant types: Authorization Code (with PKCE for SPAs/Mobile), Client Credentials.
  - OIDC: An identity layer built on top of OAuth 2.0 that provides authentication
    by introducing an `ID Token` containing user profile claims.

* Access Control Paradigms:
  - RBAC (Role-Based Access Control): Access granted based on roles (e.g., Admin, Editor).
  - ABAC (Attribute-Based Access Control): Access evaluated dynamically using attributes
    (e.g., user department, time of day, request IP, resource owner).

* Backend Security Best Practices (OWASP Top 10):
  - SQL Injection (SQLi): Prevented by using parameterized queries / prepared statements.
  - Cross-Site Scripting (XSS): Sanitizing HTML outputs, applying `HttpOnly` flags to cookies.
  - Cross-Site Request Forgery (CSRF): Enforcing `SameSite` cookie policies, anti-CSRF tokens.
  - Rate Limiting: Preventing brute force and DDoS using algorithms like Token Bucket or Leaky Bucket.
  - CORS (Cross-Origin Resource Sharing): Enforcing explicit origin whitelists rather than wildcard `*`.


4. DATABASE CONCEPTS & SYSTEM DESIGN [Importance: 9.5/10]
--------------------------------------------------------------------------------
* SQL vs. NoSQL Databases:
  - SQL (Relational): Relational schema, tabular tables, foreign key constraints.
    - Ideal for structured data, complex join operations, and strong consistency requirements.
  - NoSQL (Non-Relational): Document (MongoDB), Key-Value (Redis), Column-Family (Cassandra), Graph (Neo4j).
    - Ideal for unstructured or semi-structured data, high-velocity writing, horizontal scaling, and flexible schemas.

* ACID vs. BASE Paradigms:
  - ACID (Relational):
    - Atomicity: All operations in a transaction succeed or all fail.
    - Consistency: Data transitions from one valid state to another.
    - Isolation: Concurrent transactions do not interfere with each other.
    - Durability: Once committed, updates persist even during a crash.
  - BASE (NoSQL/Distributed):
    - Basically Available: System guarantees availability even in the face of network partitions.
    - Soft state: State of the system can change over time without user interaction.
    - Eventual consistency: System becomes consistent over time when no updates are made.

* CAP Theorem:
  In a distributed data store, you can only guarantee two of the following three:
  - Consistency (C): Every read receives the most recent write or an error.
  - Availability (A): Every non-failing node returns a response (without guarantee of latest data).
  - Partition Tolerance (P): System continues to operate despite arbitrary message loss/delay.
  - Real-world Choice: Since network partitions (P) are inevitable, distributed databases must choose
    between Consistency (CP, e.g. MongoDB, HBase) or Availability (AP, e.g. Cassandra, DynamoDB).

* Database Optimization:
  - Indexes: B-Tree / Hash tables matching search patterns. Speed up reads, slow down writes.
  - Replication: Leader-Follower replication to offload read-heavy traffic to read replicas.
  - Sharding (Horizontal Partitioning): Splitting tables horizontally across separate DB servers.
  - Connection Pooling: Reusing a pool of active database connections to avoid TCP handshake overhead.

* Repository Pattern:
  - Abstracts data access logic behind a collection-like interface, decoupling business logic from ORM frameworks or raw queries.


5. CACHING STRATEGIES [Importance: 9/10]
--------------------------------------------------------------------------------
* Why cache?
  - Reduces read latency, offloads database traffic, and increases throughput of highly repetitive reads.

* Cache-Aside (Lazy Loading):
  - Application checks cache. If miss, queries database, updates cache, and returns.
  - Pros: Only caches requested data. Cache node failure is not fatal.
  - Cons: Cache misses trigger latency, stale data if data changes directly in DB.

* Write-Through:
  - Application writes directly to the cache, which writes to the database synchronously.
  - Pros: Cache never has stale data.
  - Cons: Write latency is higher (dual-write overhead).

* Write-Behind (Write-Back):
  - Application writes to cache. Cache queue updates to database asynchronously.
  - Pros: High-throughput write performance.
  - Cons: Risk of data loss if cache fails before DB write completes.

* Common Cache Vulnerabilities:
  - Cache Stampede (Thundering Herd): High-concurrency cache key expires, causing thousands
    of requests to hit the DB at once. Mitigated using lock/semaphore or background renewal.
  - Cache Penetration: Requests for keys that do not exist (neither in cache nor DB) flood the DB.
    Mitigated by caching null values or using Bloom Filters.
  - Cache Avalanche: Multiple cached keys expire at the same time. Mitigated by adding
    random jitter/skew to expiry times.


6. MESSAGE QUEUES & EVENT-DRIVEN ARCHITECTURE [Importance: 8.5/10]
--------------------------------------------------------------------------------
* Why queues?
  - Decouples services, buffers high traffic spikes, scales consumer workers horizontally, and processes tasks asynchronously.

* Message Queues vs. Event Streams:
  - Message Queue (e.g., RabbitMQ): Message is delivered, processed, and acknowledged,
    then deleted from queue. Good for task dispatching.
  - Event Stream (e.g., Kafka): Append-only immutable log. Consumer tracks its own offset.
    Events persist and can be replayed. Excellent for event sourcing and high-throughput streaming.

* Delivery Guarantees:
  - At-least-once: Messages may be duplicated, but never lost. Consumers must be **idempotent**.
  - At-most-once: Messages are sent once; could be lost but never duplicated.
  - Exactly-once: Complex orchestration ensures every message modifies target state exactly once.


7. SCALABILITY, RESILIENCY & LOAD BALANCING [Importance: 8/10]
--------------------------------------------------------------------------------
* Horizontal vs. Vertical Scaling:
  - Vertical: Increasing resources (CPU, RAM) of a single server. Limited ceiling.
  - Horizontal: Adding more server nodes. Requires load balancing and stateless application servers.

* Load Balancer:
  - Routes traffic across multiple nodes.
  - Algorithms: Round Robin, Least Connections, IP Hash.
  - Reverse Proxy (e.g., Nginx): Client targets proxy, proxy queries backend. Handles TLS termination,
    static asset hosting, and caching.

* Resiliency Patterns:
  - Circuit Breaker: Temporarily stops forwarding requests to a failing service (tripping to Open),
    preventing cascading failures. Periodically tests downstream health (Half-Open) before closing.
  - Bulkhead Pattern: Isolates resources (thread pools, memory) to prevent one failing service component
    from consuming all system resources.


8. MONITORING, LOGGING & OBSERVABILITY [Importance: 7.5/10]
--------------------------------------------------------------------------------
* Why Logging?
  - Records execution context, trace errors, monitors system health, and audits state changes.

* Three Pillars of Observability:
  1. Metrics: Aggregate numerical values measured over time (CPU, latency, error rate).
  2. Logs: Timestamped discrete event records. Best formatted as structured JSON for easy indexing.
  3. Traces: End-to-end representation of a request's journey through a distributed microservices graph
     using correlated Trace IDs and Span IDs.

* Structured Logging vs. Unstructured Logging:
  - Unstructured Logging: Logs written as free-form strings: `[INFO] User 5 logged in from 192.168.1.1 at 12:00`. Hard to parse/search.
  - Structured Logging: Logs written in a structured format (usually JSON) containing defined fields:
    `{"timestamp": "2026-07-01T12:00:00Z", "level": "INFO", "message": "User login success", "userId": 5, "ipAddress": "192.168.1.1"}`
  - Benefit: Essential for central log aggregators (ELK, Splunk, Datadog) to index, filter, query, and trigger alerts on specific fields.

* Meaningful Error Messages & Logging Errors (Security vs. Debugging):
  - Rule of Thumb: **Never leak implementation details to the client**. Stack traces, database schema details, or raw exception messages in HTTP responses (e.g. `SQL Grammatical Error: Table 'users' not found`) can be exploited by attackers.
  - Production Pattern:
    1. Client-Side: Return clean, sanitized, standard error formats:
       `{"statusCode": 400, "errorCode": "INVALID_INPUT", "message": "The username must be alphanumeric.", "timestamp": "..."}`
    2. Backend Logs: Log the complete exception object, trace context, and stack trace internally for debugging. Link client response to internal logs using a `correlationId` or `traceId`.


9. BACKGROUND JOBS & TASK SCHEDULERS [Importance: 8/10]
--------------------------------------------------------------------------------
* Why Background Jobs?
  - Offloads resource-intensive or long-running tasks (e.g. sending emails, generating PDF reports, processing video uploads) from the HTTP request-response cycle to keep APIs responsive (avoiding 504 Gateway Timeouts).

* Core Components:
  - **Scheduler**: Triggers tasks based on time schedules (e.g. Cron jobs like `0 0 * * *` for daily midnight reports).
  - **Task/Job Queue**: A persistent message store (Redis, RabbitMQ, DB table) where jobs are queued.
  - **Worker**: Autonomous background processes/threads that poll the queue, pull jobs, and execute them.

* Retry Strategies & Retries with Exponential Backoff:
  - Background jobs can fail due to transient errors (external API timeout, database lock).
  - **Retry with Exponential Backoff**: Instead of retrying immediately (which can overload failing downstream systems - thundering herd), wait times increase exponentially with each attempt (e.g. 2s, 4s, 8s, 16s...) up to a max threshold.
  - **Jitter**: Adding random noise (jitter) to the backoff interval prevents all retried jobs from hitting downstream servers at the exact same millisecond.


10. ADVANCED DISTRIBUTED ARCHITECTURE & SYSTEM DESIGN BASICS [Importance: 8.5/10]
--------------------------------------------------------------------------------
* Distributed Transactions:
  - In microservices (database-per-service), traditional transactional locks (ACID) cannot scale.
  - **Two-Phase Commit (2PC)**: A coordinator queries all nodes to prepare writes (Phase 1), then commands them to commit (Phase 2). Highly blocking and vulnerable to coordinator crashes.
  - **Saga Pattern**: Translates a business transaction into a chain of local transactions.
    - *Choreography*: Services emit events; other services listen and execute local transactions asynchronously.
    - *Orchestration*: A central coordinator service tells each participant which transaction to execute.
    - *Compensating Transactions*: If any step fails, the Saga coordinates rollback actions in reverse to restore consistency (eventual consistency).
  - **Transactional Outbox Pattern**:
    - Ensures atomicity between database updates and message broker event publishing.
    - A local database transaction writes the update to a business table AND appends an event to an `Outbox` table.
    - An independent "Outbox Poll" process reads the Outbox table and publishes events to RabbitMQ/Kafka, ensuring at-least-once delivery without distributed locks.

* Common System Design Blueprint Components:
  - **Rate Limiter**: Controls traffic rate. Algorithms: Token Bucket (supports bursts), Leaky Bucket (smooths outflow), Sliding Window.
  - **URL Shortener**: Client sends long URL -> Server generates hash (Base62 encoding of an auto-incrementing ID or key generation service) -> Saves in DB (SQL/NoSQL) -> Returns short link. Redirects via HTTP 301 (Moved Permanently, cached) or 302 (Found, tracked).
  - **Notification Service**: Central service handling Push (Websockets/APNS/FCM), Email (SMTP/SendGrid), SMS (Twilio). Uses priority queues, templates engine, and rate limiting to avoid spamming.
  - **Chat System**: Uses WebSockets for persistent bidirectional links. WebSockets Gateway routes connections. Uses a fast database (Redis for sessions/presence, Cassandra/HBase for messaging history).
  - **File Upload System**: Client asks backend for a **Presigned URL** (S3/GCS) -> Client uploads file directly to Cloud Storage (bypasses backend to save memory) -> Cloud Storage triggers webhook to notify backend -> CDN serves files with edge caching.

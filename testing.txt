1. TESTING TERMINOLOGY: MOCKS, SPIES, STUBS, & DUMMIES [Importance: 9/10]
--------------------------------------------------------------------------------
- **Mock**: Objects pre-programmed with expectations which form a specification of the 
  calls they are expected to receive (behavior verification).
- **Spy**: A test double that records execution information (arguments, return values, call 
  frequency) of functions without modifying their actual implementation (e.g. `jest.spyOn`).
- **Stub**: Provides canned/pre-programmed responses to calls made during the test. Usually 
  does not respond to anything outside what's programmed for the test.
- **Dummy**: Objects passed around but never actually used or called. Typically used to fill 
  parameter lists.


2. UNIT TESTING: JEST vs. JASMINE / KARMA
--------------------------------------------------------------------------------
- **Jest**: A zero-config JavaScript testing framework created by Meta, commonly used in 
  React/MERN stack projects. Includes a test runner, assertion library, and mocking features.
- **Jasmine**: A behavior-driven testing library. It is the default testing framework 
  historically bundled with Angular.
- **Karma**: A test runner developed by the Angular team that launches browsers to execute 
  Jasmine tests.

* Code Syntax Comparison:
  ```javascript
  // Jest Assertion:
  expect(calculateSum(2, 3)).toBe(5);
  expect(mockFn).toHaveBeenCalledWith("data");

  // Jasmine Assertion:
  expect(calculateSum(2, 3)).toEqual(5);
  expect(spyObj.methodName).toHaveBeenCalledWith("data");
  ```


3. ANGULAR TESTBED & COMPONENT FIXTURE [Importance: 9.5/10]
--------------------------------------------------------------------------------
* Angular TestBed:
  The primary Angular testing utility. Configures and initializes an environment to instantiate 
  components/services with Dependency Injection.

* Example Component Test:
  ```typescript
  import { ComponentFixture, TestBed } from '@angular/core/testing';
  import { UsersComponent } from './users.component';
  import { UserService } from './user.service';
  import { of } from 'rxjs';

  describe('UsersComponent', () => {
    let component: UsersComponent;
    let fixture: ComponentFixture<UsersComponent>;
    let mockUserService: any;

    beforeEach(async () => {
      // 1. Create a stub/mock for dependencies
      mockUserService = jasmine.createSpyObj(['getUsers']);
      mockUserService.getUsers.and.returnValue(of([{ id: 1, name: 'Sahil' }]));

      // 2. Configure testing module
      await TestBed.configureTestingModule({
        imports: [UsersComponent], // import if standalone
        providers: [
          { provide: UserService, useValue: mockUserService }
        ]
      }).compileComponents();

      // 3. Instantiate fixture and component
      fixture = TestBed.createComponent(UsersComponent);
      component = fixture.componentInstance;
      fixture.detectChanges(); // Trigger initial lifecycle hooks (ngOnInit)
    });

    it('should load users on init', () => {
      expect(component.users.length).toBe(1);
      expect(mockUserService.getUsers).toHaveBeenCalled();
    });
  });
  ```
  - `fixture.detectChanges()`: Forces Angular to run change detection on the tested component.
  - `fixture.nativeElement`: Exposes the raw DOM element of the component for query selectors.


4. END-TO-END (E2E) TESTING: CYPRESS & PLAYWRIGHT [Importance: 8.5/10]
--------------------------------------------------------------------------------
E2E testing automates browser actions to test the entire application flow from end to end.

* Cypress:
  - Architecture: Runs inside the browser alongside your application code, yielding 
    fast execution and debugger access.
  - Pros: Auto-waiting for assertions, readable commands, time-travel debugging.
  - Cons: Restricted to executing inside a single browser tab, no multi-tab support.

* Playwright:
  - Architecture: Developed by Microsoft. Connects to browser engines (Chromium, WebKit, Firefox) 
    using the browser's native debugging protocol (WebSocket-based).
  - Pros: Native multi-page/multi-tab support, faster execution via parallel threads, 
    rich code generators.
  - Cons: Requires node engine setup, runs out-of-process.

5. BACKEND INTEGRATION TESTING [Importance: 9/10]
--------------------------------------------------------------------------------
While unit testing tests a class/function in isolation by mocking all dependencies, 
Integration Testing verifies that multiple units (e.g. Controller, Service, and 
Database/External APIs) work together correctly.

* Key Integration Testing Patterns:
  - **In-Memory / Real Databases**: Avoid testing against your production database. Instead:
    - *In-Memory Database*: Use lightweight databases (e.g., H2 database for Java/Spring Boot, sqlite memory mode) to run tests.
      - *Caveat*: SQL syntax variations (H2 behaves differently than MySQL/PostgreSQL in functions, keys, and schemas).
    - *Testcontainers (Recommended)*: A library that spins up lightweight, throwaway instances of real databases (Docker containers) during test suites execution. Ensures tests run against identical production database engines.
  - **Mocking Downstream APIs**: Use tools like MockWebServer (Java) or Nock (Node.js) to mock responses from external HTTP services instead of hitting real APIs during tests.
  - **Supertest (Node.js API Testing)**: In Express projects, `supertest` spins up the application server in memory and runs HTTP assertions:
    ```javascript
    const request = require('supertest');
    const app = require('./app');

    describe('GET /users', () => {
      it('should return a list of users', async () => {
        const res = await request(app)
          .get('/api/users')
          .set('Accept', 'application/json');
        expect(res.statusCode).toEqual(200);
        expect(res.body).toBeInstanceOf(Array);
      });
    });
    ```
  - **Transactional Rollback**: In database integration tests, wrap each test execution in a database transaction and roll it back when the test completes (`@Transactional` in Spring Boot). This ensures the test database remains clean/isolated between test cases.


6. API TESTING & POSTMAN [Importance: 8/10]
--------------------------------------------------------------------------------
Postman is a popular API client used to build, test, and document HTTP requests.

* Postman Core Testing Features:
  - **Collections**: Grouping related API requests (endpoints) for a service.
  - **Environments**: Declaring key-value variables (e.g., `{{baseUrl}}`, `{{token}}`) to easily switch between Local, QA, and Production target hosts.
  - **Chaining Requests**: Saving variables dynamically from one response to use in the next request:
    - Example: Capture JWT token on Login response and save it as a Collection Variable:
      ```javascript
      const response = pm.response.json();
      pm.collectionVariables.set("jwt_token", response.accessToken);
      ```
    - Use in subsequent headers: `Authorization: Bearer {{jwt_token}}`
  - **Postman Assertions**: Writing JavaScript assertions in the "Tests" tab of a request to automate validation:
    ```javascript
    pm.test("Status code is 200", function () {
        pm.response.to.have.status(200);
    });

    pm.test("Response contains user details", function () {
        const jsonData = pm.response.json();
        pm.expect(jsonData.name).to.eql("Sahil");
        pm.expect(jsonData.role).to.exist;
    });
    ```
  - **Newman (CLI Runner)**: Command-line runner for Postman collections, allowing you to run Postman collections in CI/CD pipelines: `newman run my_collection.json -e my_env.json`.

⭐⭐⭐⭐⭐ NODE.JS BASICS

1. What is Node.js?
   - **Answer**: Node.js is an open-source, cross-platform JavaScript runtime environment built on Google Chrome's V8 JavaScript engine. It allows developers to run JavaScript on the server side, outside the browser, enabling the creation of scalable, high-performance network applications.

2. Why Node.js is used?
   - **Answer**: Node.js is primarily used for building fast, scalable network applications, such as RESTful APIs, real-time chat applications, streaming services, and single-page application backends. It is chosen for:
     - High performance (V8 Engine)
     - Non-blocking, asynchronous execution (ideal for I/O-intensive apps)
     - Single programming language (JavaScript/TypeScript) across the entire stack (front-end and back-end)
     - Massive package ecosystem (npm)

3. Is Node.js single-threaded or multi-threaded?
   - **Answer**: Node.js is fundamentally **single-threaded** for executing JavaScript code on the main thread (using the Event Loop). However, it is **multi-threaded** under the hood. For heavy operations like file system access (fs), cryptography (crypto), DNS lookups, and network operations, Node.js delegates tasks to a pool of worker threads managed by the C-based auxiliary library **libuv** (known as the Thread Pool).

4. Explain Event Loop.
   - **Answer**: The Event Loop is the core mechanism that allows Node.js to perform non-blocking I/O operations despite being single-threaded. It constantly monitors the Call Stack and the Callback Queue. When the Call Stack is empty, the Event Loop takes tasks from the queue and pushes them onto the Call Stack to be executed.
   - It operates in distinct phases:
     1. **Poll**: Retrieves new I/O events.
     2. **Check**: Executes callbacks registered via `setImmediate()`.
     3. **Close callbacks**: Executes close event callbacks (e.g., `socket.on('close')`).
     4. **Timers**: Executes callbacks scheduled by `setTimeout()` and `setInterval()`.
     5. **Pending callbacks**: Executes I/O callbacks deferred from the previous loop iteration.
     6. **Idle, prepare**: Used internally only.

5. What is non-blocking I/O?
   - **Answer**: Non-blocking I/O means that database operations, network requests, or file read/write operations do not stop (block) the execution of the main thread. Instead of waiting for a resource to return data, Node.js registers a callback (or returns a Promise) and moves on to execute subsequent code. Once the I/O operation is completed in the background, the callback is queued for execution.

6. Difference between synchronous and asynchronous programming.
   - **Answer**:
     - **Synchronous**: Code is executed line-by-line sequentially. Each operation must finish before the next one starts. This can block the application if an operation takes a long time.
     - **Asynchronous**: Code starts an operation and moves to the next line immediately. When the async operation completes, its callback/promise handler is executed, preventing blocking and improving responsiveness.

7. What are callbacks?
   - **Answer**: A callback is a function passed as an argument to another function, which is intended to be executed after the outer function has completed its execution. They are the traditional way Node.js handled asynchronous actions.

8. What are Promises?
   - **Answer**: A Promise is an object representing the eventual completion (or failure) of an asynchronous operation and its resulting value. It exists in one of three states:
     - `Pending`: Initial state, neither fulfilled nor rejected.
     - `Fulfilled`: The operation completed successfully.
     - `Rejected`: The operation failed with an error.

9. async/await vs Promises.
   - **Answer**:
     - `async/await` is a syntactic sugar built on top of Promises.
     - It allows asynchronous code to be written in a synchronous-looking style, making it cleaner and easier to read/debug.
     - Error handling in `async/await` is done using standard `try/catch` blocks, whereas Promises use `.then()` and `.catch()`.

10. What is package.json?
    - **Answer**: `package.json` is the manifest file of a Node.js project. It contains metadata about the project (name, version, description, entry point) and lists the dependencies (packages required for production) and devDependencies (packages required only for development/testing), along with scripts to run commands.

11. What is package-lock.json?
    - **Answer**: `package-lock.json` is automatically generated by npm whenever dependencies are installed or updated. It records the exact version of every installed package and its sub-dependencies, ensuring that the dependency tree is completely identical across different environments, machines, and deployments.

12. Difference between CommonJS and ES Modules.
    - **Answer**:
      - **CommonJS (CJS)**: Uses `require()` to import modules and `module.exports` to export. It loads modules synchronously and is the default module system in older Node.js.
      - **ES Modules (ESM)**: Uses `import` and `export` statements. It supports static analysis, loads modules asynchronously, and is the standard JS module specification. Enabled in Node.js via `"type": "module"` in `package.json` or `.mjs` extensions.

13. require() vs import.
    - **Answer**:
      - `require()`: Part of CommonJS; evaluated at runtime (dynamic imports are easy); synchronous; cannot be used in standard ES Modules directly.
      - `import`: Part of ES Modules; statically analyzed and resolved at compile/parse time (enabling tree-shaking); asynchronous; must be at the top-level of a file (unless using dynamic `import()`).

14. What is npm?
    - **Answer**: npm (Node Package Manager) is the default package manager for the Node.js runtime environment. It consists of a command-line client and an online registry of public and private packages. It is used to install, share, and manage project dependencies.

15. What is npx?
    - **Answer**: npx (Node Package Executable) is a tool that comes bundled with npm (version 5.2+). It allows developers to execute packages from the npm registry directly without installing them globally or locally in the project (e.g., `npx create-react-app my-app`).

16. What are environment variables (.env)?
    - **Answer**: Environment variables are key-value pairs configured outside the application code, typically stored in a `.env` file during development. They are used to keep sensitive configuration (like database credentials, API keys, and port numbers) separate from source code, preventing security leaks.

17. What is process.env?
    - **Answer**: `process.env` is a global object in Node.js provided by the runtime environment. It exposes the user environment variables to the active Node.js process (e.g., `process.env.PORT` or `process.env.DB_URL`).

18. What is EventEmitter?
    - **Answer**: `EventEmitter` is a core class in Node.js's `events` module. It facilitates communication between objects by allowing an object to emit named events that cause registered listeners (functions) to be called. It is the basis for much of Node's built-in asynchronous architecture (like streams and sockets).

19. What are Streams?
    - **Answer**: Streams are collection pipelines that let you read data from a source or write data to a destination continuously, chunk by chunk, without loading the entire dataset into memory. This is highly memory-efficient for processing large files or network packets. Types of streams: Readable, Writable, Duplex, and Transform.

20. What are Buffers?
    - **Answer**: A Buffer is a temporary storage area in memory used to store raw binary data. It represents a fixed-size chunk of memory allocated outside the V8 heap. Buffers are commonly used when dealing with TCP streams, file system operations, or binary protocols.

⭐⭐⭐⭐⭐ EXPRESS.JS

21. What is Express.js?
    - **Answer**: Express.js is a minimal, flexible Node.js web application framework that provides a robust set of features for building web and mobile applications, specifically facilitating routing, middleware integration, and HTTP request/response handling.

22. Why Express instead of pure Node.js?
    - **Answer**: While pure Node.js (using the `http` module) requires writing verbose boilerplates for routing, parsing headers, serving static files, and handling errors, Express simplifies these workflows by providing high-level abstractions, routing tables, and middleware chains, speeding up development significantly.

23. What is express()?
    - **Answer**: `express()` is a top-level function exported by the Express module. Calling it instantiates an Express application object (usually assigned to `app`), which provides methods to configure routing, middleware, settings, and start the HTTP server.

24. What is app.listen()?
    - **Answer**: `app.listen()` is a method used to start the Express application on a specified port and host. It creates an HTTP server under the hood using Node's native `http.createServer()` and binds the Express application as the request listener.

25. What is app.use()?
    - **Answer**: `app.use()` is used to register middleware functions globally in the Express application middleware stack. Any request entering the application will pass through the middlewares registered via `app.use()` in the order they are defined.

26. What is express.Router()?
    - **Answer**: `express.Router()` creates a modular, mini-Express app capable of performing middleware and routing functions. It is used to separate route definitions into separate files (e.g., `userRoutes.js`, `productRoutes.js`), making large-scale projects manageable.

27. What is Middleware?
    - **Answer**: Middleware is a function that has access to the Request object (`req`), the Response object (`res`), and the `next` middleware function in the application’s request-response cycle. It can execute code, modify request and response objects, end the request-response cycle, or call the next middleware.

28. Types of Middleware.
    - **Answer**:
      - **Application-level Middleware**: Bound to the app object using `app.use()` or `app.METHOD()`.
      - **Router-level Middleware**: Bound to an instance of `express.Router()`.
      - **Error-handling Middleware**: Middleware defined with four arguments `(err, req, res, next)`.
      - **Built-in Middleware**: Provided by Express (e.g., `express.json()`, `express.static()`).
      - **Third-party Middleware**: Installed via npm (e.g., `cors`, `helmet`, `morgan`).

29. What is next()?
    - **Answer**: `next()` is a function callback passed to a middleware. When invoked, it passes control to the next middleware function in the stack. If `next()` is not called and the response is not ended, the client request will hang.

30. Difference between middleware and controller.
    - **Answer**:
      - **Middleware**: Intercepts the request before it reaches the final handler. Used for cross-cutting concerns like logging, authentication, request validation, or header injection.
      - **Controller**: The final endpoint handler that contains the business logic for a specific route. It processes the request and sends the final response back to the client.

31. How does Express process a request?
    - **Answer**: When a request hits the Express server:
      1. It matches the HTTP method and path.
      2. It executes registered global middlewares in order.
      3. It executes router-specific middlewares.
      4. It matches the route handler (controller) and processes the business logic.
      5. The controller sends the response (using `res.send()`, `res.json()`, etc.), ending the cycle.

32. How do you organize routes in Express?
    - **Answer**: Routes are organized using `express.Router()`. A typical directory structure has a `routes/` folder containing files like `authRoutes.js`, `userRoutes.js`, which are then imported and mounted in the main server file (`index.js` or `app.js`) using `app.use('/api/users', userRoutes)`.

33. How do you serve static files?
    - **Answer**: Static assets (images, CSS, JS files, HTML files) are served using Express's built-in `express.static` middleware.
      - Example: `app.use(express.static('public'));` or `app.use('/static', express.static(path.join(__dirname, 'public')));`.

34. What is express.json()?
    - **Answer**: It is a built-in middleware function in Express that parses incoming requests with JSON payloads and makes the parsed data available under `req.body`.

35. What is express.urlencoded()?
    - **Answer**: It is a built-in middleware function in Express that parses incoming requests with URL-encoded payloads (typically submitted from HTML forms) and makes the parsed data available under `req.body`.

⭐⭐⭐⭐⭐ REST APIs

36. What is REST?
    - **Answer**: REST (Representational State Transfer) is an architectural style for designing networked applications. It relies on a stateless, client-server, cacheable communications protocol, almost always using HTTP. RESTful APIs use HTTP methods (GET, POST, PUT, DELETE) to perform CRUD operations on resources identified by URIs.

37. What is CRUD?
    - **Answer**: CRUD stands for Create, Read, Update, and Delete. These are the four basic functions of persistent storage, mapped in REST APIs to:
      - Create -> `POST`
      - Read -> `GET`
      - Update -> `PUT` / `PATCH`
      - Delete -> `DELETE`

38. Difference between GET and POST.
    - **Answer**:
      - **GET**: Used to retrieve data from a server. Parameters are appended to the URL query string. It is idempotent, cacheable, and has length restrictions.
      - **POST**: Used to send data to the server to create/modify a resource. Parameters are sent in the HTTP request body. It is non-idempotent, not cacheable by default, and has no size restrictions.

39. Difference between PUT and PATCH.
    - **Answer**:
      - **PUT**: Replaces the entire target resource with the new representation provided in the request payload (idempotent).
      - **PATCH**: Applies partial modifications to a resource, updating only the specific fields provided in the payload (non-idempotent by specification, though often implemented as idempotent).

40. Difference between DELETE and PATCH.
    - **Answer**:
      - **DELETE**: Removes the specified resource entirely.
      - **PATCH**: Modifies properties of an existing resource without deleting it.

41. Path Parameters vs Query Parameters.
    - **Answer**:
      - **Path Parameters**: Part of the URI path used to identify a specific resource (e.g., `/api/users/:id` -> `/api/users/123`).
      - **Query Parameters**: Key-value pairs appended after `?` in the URL, used for filtering, sorting, or paginating resources (e.g., `/api/users?status=active&sort=desc`).

42. Request Body vs Query Params.
    - **Answer**:
      - **Request Body**: Used to send complex, structured data payloads (usually JSON) securely inside the HTTP request. Typically used in `POST`, `PUT`, and `PATCH`.
      - **Query Params**: Short string key-value pairs exposed in the URL, visible to the user and saved in browser history. Typically used in `GET` requests.

43. Common HTTP Status Codes.
    - **Answer**:
      - **200 OK**: Request succeeded.
      - **201 Created**: Resource successfully created.
      - **400 Bad Request**: Client sent invalid data.
      - **401 Unauthorized**: Authentication is required or has failed.
      - **403 Forbidden**: Client is authenticated but lacks permission to access the resource.
      - **404 Not Found**: Resource could not be found.
      - **500 Internal Server Error**: Generic server-side error.

44. Idempotent APIs.
    - **Answer**: An API method is idempotent if making multiple identical requests has the same effect on the server state as making a single request. `GET`, `PUT`, `DELETE`, `HEAD`, and `OPTIONS` are idempotent. `POST` and `PATCH` are non-idempotent.

45. REST API naming best practices.
    - **Answer**:
      - Use plural nouns for resources (e.g., `/api/employees`, not `/api/employee`).
      - Use lowercase letters and separate words with hyphens (kebab-case) (e.g., `/api/employee-records`).
      - Do not include verbs in the path; map actions to HTTP methods (e.g., use `DELETE /api/users/123`, not `/api/users/delete/123`).
      - Use nesting to show relationships (e.g., `/api/departments/5/employees`).

⭐⭐⭐⭐⭐ AUTHENTICATION

46. Authentication vs Authorization.
    - **Answer**:
      - **Authentication**: Verifying who a user is (e.g., verifying username/password or token).
      - **Authorization**: Verifying what access permissions the authenticated user has (e.g., checking if the user is an 'admin' or 'employee').

47. What is JWT?
    - **Answer**: JWT (JSON Web Token) is a compact, URL-safe means of representing claims to be transferred between two parties. It is digitally signed, meaning the server can verify the integrity of the claims without holding state or querying a database on every request.

48. Structure of JWT.
    - **Answer**: A JWT is composed of three parts separated by dots (`.`):
      1. **Header**: Contains metadata about the token type (JWT) and signing algorithm (e.g., HS256, RS256).
      2. **Payload**: Contains the claims (user data like `userId`, `role`, and expiration time `exp`).
      3. **Signature**: Formed by encoding the header and payload and signing them using a secret key.

49. How JWT authentication works.
    - **Answer**:
      1. Client sends credentials (username/password) to the server.
      2. Server validates credentials and generates a signed JWT.
      3. Server returns the JWT to the client.
      4. Client stores the JWT (local storage or cookie) and sends it in the `Authorization` header (`Bearer <token>`) for subsequent requests.
      5. Server verifies the signature of the incoming JWT. If valid, the request is processed.

50. Why password hashing is required.
    - **Answer**: Storing raw, plain-text passwords in a database is a major security risk. If the database is compromised, all user accounts are exposed. Hashing converts passwords into an irreversible string of fixed length, ensuring that even if data is stolen, the raw passwords cannot be decrypted.

51. bcrypt vs encryption.
    - **Answer**:
      - **bcrypt**: A one-way hashing function designed specifically for passwords. It is slow and uses a salt to protect against brute-force and rainbow table attacks. It cannot be reversed (decrypted).
      - **Encryption**: A two-way process that transforms plain-text into cipher-text and can be decrypted back to plain-text using a secret key.

52. Access Token vs Refresh Token.
    - **Answer**:
      - **Access Token**: Short-lived (e.g., 15 mins) token sent with every API request to authorize access.
      - **Refresh Token**: Long-lived (e.g., 7 days) token stored securely, used solely to request a new Access Token when the current one expires, reducing security exposure if the access token is intercepted.

53. Where should JWT be stored?
    - **Answer**:
      - **HttpOnly Secure Cookies**: Recommended. They are immune to Cross-Site Scripting (XSS) attacks because JavaScript cannot access them.
      - **LocalStorage / SessionStorage**: Vulnerable to XSS. If a malicious script runs on your page, it can read the token.

54. Cookies vs Local Storage.
    - **Answer**:
      - **Cookies**: Automatically sent by the browser with every HTTP request to the matching domain. Supported size is small (~4KB). Can be protected with `HttpOnly` and `Secure` flags.
      - **Local Storage**: Persistent storage in the browser, accessible via JavaScript. Storage capacity is larger (~5-10MB). Not sent automatically with requests.

55. HttpOnly Cookies.
    - **Answer**: A flag set on a cookie that prevents client-side scripts (JavaScript) from accessing it. This mitigates the risk of session hijacking via Cross-Site Scripting (XSS) attacks.

⭐⭐⭐⭐⭐ ERROR HANDLING

56. try-catch.
    - **Answer**: A control-flow construct used to handle exceptions. Code that might throw an error is wrapped in the `try` block, and the error is handled gracefully inside the `catch` block without crashing the Node.js application.

57. Centralized Error Handling.
    - **Answer**: A design pattern where all errors thrown anywhere in the application are caught and sent to a single location (a centralized error-handling middleware in Express) to format the response and log the details consistently.

58. Error Middleware.
    - **Answer**: A special type of middleware in Express defined with exactly four parameters: `(err, req, res, next)`. Express recognizes it by the arity (four arguments) and delegates error handling to it when `next(err)` is called.

59. next(error).
    - **Answer**: Passing any argument (except the string `'route'`) to `next()` tells Express that the current request resulted in an error. Express will skip all remaining non-error-handling middlewares and routes and jump directly to the registered error-handling middleware.

60. Custom Error Classes.
    - **Answer**: Extending JavaScript's built-in `Error` class to create application-specific errors (e.g., `class AppError extends Error`). This allows storing additional metadata like `statusCode` and handling them dynamically in the centralized error handler.

61. Handling async errors.
    - **Answer**: In older Express versions, unhandled rejections inside async functions crashed the process or hung the request. To handle async errors:
      - Wrap block in `try-catch` and call `next(err)`.
      - Use a wrapper utility (like `express-async-errors` library or a custom `catchAsync` wrapper function) to automatically forward rejected promises to `next()`.

⭐⭐⭐⭐⭐ DATABASE

62. SQL vs NoSQL.
    - **Answer**:
      - **SQL (Relational)**: Structured, table-based schema with relations. Uses SQL for queries. ACID compliant. Scaled vertically. Examples: PostgreSQL, MySQL.
      - **NoSQL (Non-Relational)**: Flexible, schema-less structures (Document, Key-Value, Graph). Scaled horizontally. BASE consistency model. Examples: MongoDB, Redis.

63. MySQL vs MongoDB.
    - **Answer**:
      - **MySQL**: Relational Database Management System (RDBMS). Data stored in tables with rows and columns. Excellent for complex joins, strict schemas, and transactions.
      - **MongoDB**: Document database. Data stored in JSON-like BSON documents. High write speed, scalable, schema flexibility, but joins are less performant (uses `$lookup`).

64. Primary Key.
    - **Answer**: A column (or set of columns) in a relational database table that uniquely identifies each row in that table. It must contain unique values and cannot contain NULL values.

65. Foreign Key.
    - **Answer**: A column or group of columns in a relational table that provides a link between data in two tables. It references the Primary Key of another table, ensuring referential integrity.

66. Index.
    - **Answer**: A database optimization structure that improves the speed of data retrieval operations on a table at the cost of additional storage space and slower writes (insert/update).

67. Joins.
    - **Answer**: SQL operations used to combine rows from two or more tables based on a related column between them. Types: `INNER JOIN`, `LEFT JOIN`, `RIGHT JOIN`, `FULL JOIN`.

68. Connection Pool.
    - **Answer**: A cache of database connections maintained by the backend application so that connections can be reused when queries need to be executed. Reusing existing connections avoids the high overhead of establishing a new connection for every request.

69. Transactions.
    - **Answer**: A database transaction is a sequence of one or more SQL operations treated as a single unit of work. Transactions ensure data integrity by guaranteeing that either all operations succeed and are saved (committed) or none do, rolling back the database to its previous state if any operation fails.

70. ACID Properties.
    - **Answer**:
      - **Atomicity**: All operations succeed or fail together.
      - **Consistency**: Data must transition from one valid state to another.
      - **Isolation**: Concurrent transactions do not interfere with each other.
      - **Durability**: Committed data survives system crashes.

71. Normalization.
    - **Answer**: The process of structuring a relational database to reduce data redundancy and improve data integrity by splitting tables and defining relationships between them (e.g., 1NF, 2NF, 3NF).

72. Pagination.
    - **Answer**: The process of dividing a large dataset into discrete, smaller chunks (pages). Implemented using SQL `LIMIT` and `OFFSET` or cursor-based pagination.

73. Sorting.
    - **Answer**: Ordering the result set based on a specific field in ascending (`ASC`) or descending (`DESC`) order.

74. Filtering.
    - **Answer**: Restricting the rows returned from the database based on search criteria or conditions using SQL `WHERE` clauses.

⭐⭐⭐⭐⭐ ARCHITECTURE

75. MVC Architecture.
    - **Answer**: MVC (Model-View-Controller) is a design pattern used to separate an application into three interconnected components:
      - **Model**: Manages data and business rules (database schemas).
      - **View**: The presentation layer (in Node APIs, this is the JSON response, or handled by Angular on the front-end).
      - **Controller**: Intercepts requests, calls services, and return responses.

76. Controller.
    - **Answer**: In REST APIs, the Controller handles incoming HTTP requests, extracts parameters/body data, interacts with the Service Layer, and sends the appropriate HTTP response back to the client.

77. Service Layer.
    - **Answer**: A design pattern where the business logic of the application is isolated from HTTP handling or database querying. The Controller calls methods in the Service layer to execute business rules.

78. Repository Pattern.
    - **Answer**: A design pattern that abstracts data access. It acts as a mediator between the domain/business logic (Service layer) and data mapping layers (Database ORMs like Mongoose or Sequelize).

79. Why separate Controller and Service?
    - **Answer**: It enforces separation of concerns:
      - Makes code highly reusable (e.g., the same business service can be invoked by an HTTP controller, a CLI command, or a cron job).
      - Facilitates easier unit testing because business logic can be tested independently of Express req/res objects.

80. Folder Structure.
    - **Answer**: A typical production folder structure:
      ```text
      src/
      ├── config/         # Db connection, env files
      ├── controllers/    # Route handlers
      ├── services/       # Business logic
      ├── models/         # Database models/schemas
      ├── routes/         # Routing files
      ├── middlewares/    # Custom middlewares (auth, validation)
      ├── utils/          # Helper files
      └── app.js          # Entry point
      ```

81. Dependency Injection (concept).
    - **Answer**: Dependency Injection (DI) is a software design pattern where an object receives other objects (dependencies) it requires, rather than creating them internally. This makes testing easier since mock dependencies can be injected.

⭐⭐⭐⭐⭐ SECURITY

82. CORS.
    - **Answer**: CORS (Cross-Origin Resource Sharing) is a browser security mechanism that uses HTTP headers to tell browsers whether to allow a web application running at one origin to access resources from a different origin. In Express, it is configured using the `cors` middleware.

83. Helmet.
    - **Answer**: Helmet is a security middleware for Express that helps secure the application by setting various HTTP response headers (e.g., disabling `X-Powered-By`, setting `X-Frame-Options` to prevent Clickjacking).

84. SQL Injection.
    - **Answer**: A security vulnerability where an attacker manipulates SQL queries by injecting malicious input. Prevented by using parameterized queries (prepared statements) or ORMs, rather than concatenating user input directly into SQL strings.

85. XSS.
    - **Answer**: Cross-Site Scripting (XSS) occurs when malicious client-side script is injected into trusted websites. Prevented by sanitizing user input, escaping output, and using `HttpOnly` flags for cookies.

86. CSRF.
    - **Answer**: Cross-Site Request Forgery (CSRF) forces an authenticated user to execute unwanted actions on a web application where they are currently authenticated. Prevented using anti-CSRF tokens or using `SameSite` cookie attributes.

87. Rate Limiting.
    - **Answer**: Restricting the number of requests a user or IP address can make to the server in a given timeframe (preventing DDoS attacks and brute-forcing). Implemented using packages like `express-rate-limit`.

88. Input Validation.
    - **Answer**: The practice of checking if input matches defined constraints (e.g., type, length, email format) before processing. Usually done using validation schemas like `Joi` or `express-validator`.

89. Sanitization.
    - **Answer**: The process of cleaning user input to remove harmful data (like script tags or dangerous characters) before storing or rendering it, mitigating XSS and injection risks.

⭐⭐⭐⭐⭐ PERFORMANCE

90. Caching.
    - **Answer**: Storing copies of frequently accessed data in a fast temporary storage layer (like memory cache or Redis) to resolve future requests faster and reduce database load.

91. Compression.
    - **Answer**: Reducing the size of HTTP response bodies before sending them to the client (typically via Gzip/Brotli algorithms), saving network bandwidth.

92. Gzip.
    - **Answer**: A compression algorithm supported by most modern web browsers. In Express, the `compression` middleware is used to compress response bodies.

93. Lazy Loading.
    - **Answer**: A design pattern that defers the initialization/loading of a resource or module until the point at which it is actually needed, optimizing start-up performance.

94. Connection Pooling.
    - **Answer**: Managing database connection resources efficiently by retaining a collection of active connections that can be reused, avoiding the overhead of constantly opening and closing connections.

95. Pagination for large datasets.
    - **Answer**: Implementing pagination limit-offset queries or key-based (cursor) pagination. Cursor-based pagination is preferred for large datasets because offset-based queries degrade in performance as the offset increases.

⭐⭐⭐⭐⭐ FILES

96. fs module.
    - **Answer**: The built-in File System (`fs`) module in Node.js provides APIs to interact with the file system (create, read, update, delete files and directories).

97. readFile vs readFileSync.
    - **Answer**:
      - `readFile`: Asynchronous; does not block the Event Loop while reading the file. Operates via callbacks or promises.
      - `readFileSync`: Synchronous; blocks execution of all other tasks on the thread until the entire file is read.

98. Streams vs readFile.
    - **Answer**:
      - `readFile` reads the entire file content into memory at once before returning it. It will fail or consume high RAM for very large files.
      - `Streams` read/write files in small chunks sequentially, allowing processing of files larger than the available physical RAM.

99. Uploading files using Multer.
    - **Answer**: Multer is an Express middleware for handling `multipart/form-data`, which is primarily used for uploading files. It parses request fields and handles saving files to disk or storage.

100. Serving static files.
     - **Answer**: Handled by Express using the built-in `express.static` middleware, serving files like HTML, CSS, and JS directly from the server.

⭐⭐⭐⭐⭐ ASYNCHRONOUS PROGRAMMING

101. Callback Hell.
     - **Answer**: A situation where multiple nested callbacks make the code deeply nested, hard to read, and difficult to maintain (also known as the "Pyramid of Doom").

102. Promise Chaining.
     - **Answer**: A pattern where multiple asynchronous operations are executed sequentially by returning a new Promise from a `.then()` handler, allowing flat, readable async workflows.

103. async/await.
     - **Answer**: A clean, modern way to handle promises in JavaScript. It wraps promise-based asynchronous execution in code that reads as if it were synchronous.

104. Promise.all().
     - **Answer**: Takes an array of Promises and returns a single Promise that resolves when all input promises have resolved. If any promise rejects, the entire returned promise rejects immediately (fail-fast).

105. Promise.allSettled().
     - **Answer**: Takes an array of Promises and returns a promise that resolves after all input promises have completed (either fulfilled or rejected), providing status and outcome details for each.

106. Promise.race().
     - **Answer**: Takes an array of Promises and returns a promise that resolves or rejects as soon as *one* of the input promises resolves or rejects.

⭐⭐⭐⭐⭐ NODE INTERNALS

107. Event Loop Phases.
     - **Answer**: The phases execute in a loop: Timers -> Pending Callbacks -> Idle/Prepare -> Poll -> Check -> Close Callbacks.

108. Microtask Queue.
     - **Answer**: A queue that holds callbacks of `process.nextTick` and Promise handlers (then/catch/finally). Microtask tasks are executed immediately after the current operation finishes, before the next phase of the event loop.

109. Callback Queue.
     - **Answer**: (Also known as Macrotask Queue) holds callbacks from I/O events, timers (`setTimeout`), etc. Tasks here run after all microtasks have been completed.

110. process.nextTick().
     - **Answer**: A function that schedules a callback to be invoked immediately at the end of the current operation, before the Event Loop continues to the next phase or executes any other microtask.

111. setTimeout().
     - **Answer**: Schedules a callback to be run after a minimum delay in milliseconds. The callback goes to the Timers phase macrotask queue.

112. setImmediate().
     - **Answer**: Schedules a callback to run immediately in the Check phase of the Event Loop, after the Poll phase completes.

113. Worker Threads.
     - **Answer**: The `worker_threads` module enables the execution of CPU-intensive JavaScript code in parallel threads, helping to prevent CPU-bound tasks from blocking the main Event Loop.

114. Cluster.
     - **Answer**: A core module that allows spawning child processes (workers) that share the same server port. This leverages multi-core CPUs by running multiple instances of the Node application.

115. libuv.
     - **Answer**: A multi-platform C support library that gives Node.js its asynchronous, non-blocking I/O capability. It implements the Event Loop and the thread pool for operations.

⭐⭐⭐⭐⭐ SCENARIO QUESTIONS

116. Design Login API.
     - **Answer**:
       1. Receive `email` and `password` in `req.body`.
       2. Validate inputs.
       3. Fetch user record from database by email.
       4. Compare submitted password with hashed password in database using `bcrypt.compare()`.
       5. If correct, generate a JWT.
       6. Send JWT back in HttpOnly cookie or response body.

117. Design Register API.
     - **Answer**:
       1. Receive user details (`name`, `email`, `password`) in `req.body`.
       2. Validate input and check if user already exists.
       3. Hash password using `bcrypt.hash(password, saltRounds)`.
       4. Save new user record to database.
       5. Return success status (201 Created) along with a success message or verification email flow.

118. Design Employee CRUD APIs.
     - **Answer**:
       - `POST /api/employees`: Create new employee (validates data, saves to DB).
       - `GET /api/employees`: Read list of employees (supports pagination, filtering).
       - `GET /api/employees/:id`: Read details of a specific employee.
       - `PUT /api/employees/:id`: Update all fields of an employee.
       - `DELETE /api/employees/:id`: Remove an employee.

119. How Angular communicates with Express backend.
     - **Answer**: Angular uses its `HttpClient` module to make HTTP requests (GET, POST, etc.) to the Express backend endpoints. To facilitate local development, standard Angular configurations use a `proxy.conf.json` file to route front-end requests from `localhost:4200` to the Express backend port, preventing CORS errors.

120. Complete Request Lifecycle.
     - **Answer**:
       1. Client makes an HTTP request.
       2. Request reaches Server, parsed by middleware (e.g., `express.json()`).
       3. Cross-origin checks performed (CORS).
       4. Authenticated using Auth Middleware.
       5. Matched Route Handler executes logic.
       6. Database queries run.
       7. Response returned (e.g., 200 OK with JSON data).

121. How authentication middleware works.
     - **Answer**:
       1. Extracts token from `Authorization` header (`Bearer <token>`) or cookies.
       2. If token is missing, returns `401 Unauthorized`.
       3. Verifies token using `jwt.verify()`.
       4. If valid, attaches the decoded payload (e.g., `req.user = decoded`) and calls `next()`.
       5. If invalid/expired, returns `401 Unauthorized`.

122. How centralized error handling works.
     - **Answer**:
       - Controllers wrap async code inside try-catch and forward errors using `next(err)`.
       - An error handling middleware is defined at the end of the middleware stack: `app.use((err, req, res, next) => { res.status(err.statusCode || 500).json({ message: err.message }); })`.

123. How JWT verification works.
     - **Answer**:
       - The server decodes the token header and payload.
       - It hashes the header and payload using the server's private secret key.
       - It compares this new signature against the signature attached to the incoming JWT. If they match and the token expiration `exp` hasn't passed, verification succeeds.

124. How pagination works.
     - **Answer**:
       - Client sends `page` and `limit` as query params.
       - Server calculates `offset = (page - 1) * limit`.
       - Database query uses `LIMIT limit OFFSET offset` (in SQL) or `.skip(offset).limit(limit)` (in Mongoose).
       - Returns the records along with pagination metadata (currentPage, totalPages, totalCount).

125. How search + sorting APIs are implemented.
     - **Answer**:
       - Client sends parameters (e.g., `/employees?search=john&sort=name&order=asc`).
       - Server maps `search` parameter to database search condition (e.g., `LIKE %john%` or Regex).
       - Maps `sort` and `order` directly to database order-by statements (e.g., `.sort({ [sortField]: orderDirection })`).

⭐⭐⭐⭐⭐ GIT + DEPLOYMENT

126. Git Merge.
     - **Answer**: Merging integrates changes from a source branch into a target branch by creating a "merge commit" that ties the histories of both branches together.

127. Git Rebase.
     - **Answer**: Rebasing moves or combines a sequence of commits onto a new base commit. It rewrites project history by placing your branch's commits on top of the latest target branch commits, creating a clean linear history.

128. Merge Conflicts.
     - **Answer**: Occurs when changes are made on the same lines of the same file in different branches, and Git cannot automatically determine which changes to keep. Developers must resolve these conflicts manually before completing the merge or rebase.

129. Pull Request Process.
     - **Answer**:
       1. Push local feature branch to remote.
       2. Open a Pull Request (PR) in repository host (GitHub, GitLab).
       3. Trigger automated tests and builds (CI pipeline).
       4. Request review from teammates.
       5. Address feedback and merge branch to main branch.

130. Code Review.
     - **Answer**: A process where team members inspect code updates in a pull request to ensure technical accuracy, check for security flaws, maintain styling guidelines, and verify that requirements are met.

131. CI/CD Basics.
     - **Answer**:
       - **Continuous Integration (CI)**: Automatically building and testing code changes whenever developers push to the repository.
       - **Continuous Delivery/Deployment (CD)**: Automatically releasing/deploying the validated build artifact to staging or production environments.

⭐⭐⭐⭐⭐ ORM & VALIDATION LIBRARIES

132. What is an ORM and ODM?
     - **Answer**:
       - **ORM (Object-Relational Mapping)**: A technique/tool that lets you query and manipulate data from a relational database (like MySQL, PostgreSQL) using an object-oriented paradigm in your programming language. Examples: Sequelize, Prisma, TypeORM.
       - **ODM (Object-Document Mapping)**: Similar to ORM, but specifically designed for Document-oriented NoSQL databases (like MongoDB) to map documents to JavaScript objects. Example: Mongoose.

133. What are the pros and cons of using an ORM?
     - **Answer**:
       - **Pros**:
         - Speeds up development (no need to write raw SQL).
         - Reduces boilerplate code and automatically maps data to objects.
         - Abstracted database driver (makes switching databases easier).
         - Built-in security features (automatically parameterizes queries to prevent SQL Injection).
         - Supports database migrations.
       - **Cons**:
         - Performance overhead (slower than raw SQL queries for complex operations).
         - Learning curve for complex queries.
         - "Object-Relational Impedance Mismatch" (mapping relational tables to objects can be complex).

134. Compare Sequelize, Mongoose, and Prisma.
     - **Answer**:
       - **Mongoose**: The standard ODM for MongoDB. Offers schema definition, built-in validation, and middleware (hooks) support. Very mature and widely used.
       - **Sequelize**: A traditional, promise-based ORM for SQL databases (MySQL, Postgres, SQLite). Uses JavaScript classes to define models and relationships. Good, but can be verbose.
       - **Prisma**: A next-generation ORM for SQL and MongoDB. Uses a custom schema file (`schema.prisma`) to auto-generate a type-safe client. Offers migrations out of the box and is highly preferred in TypeScript environments.

135. What is Zod and how does it compare to Joi/Yup?
     - **Answer**:
       - **Zod**: A TypeScript-first schema declaration and validation library. It is designed to automatically infer static types from schemas, eliminating the need to duplicate types/schemas in TypeScript projects.
       - **Joi / Yup**: Older validation libraries. While powerful, they are JavaScript-first and require manual type declarations or auxiliary libraries to get type safety in TypeScript. Zod has largely replaced Joi/Yup in modern Node/TypeScript setups.

136. What is express-validator and how does it differ from schema-based validation?
     - **Answer**:
       - **express-validator**: A middleware-based validation library wrapper around `validator.js`. It runs validators directly on the Express request object (e.g., `body('email').isEmail()`) in the routing middleware chain.
       - **Schema validation (Zod/Joi)**: Defines a separate standalone data contract/schema, and validates input objects against it. It is decoupled from Express, making it easier to reuse validation logic in services or other parts of the application.

137. How do you handle database migrations in an ORM?
     - **Answer**: Database migrations act as version control for your database schema.
       - An ORM provides a CLI tool (e.g., Sequelize CLI, Prisma Migrate) to generate migration files containing `up` (to apply changes) and `down` (to revert changes) scripts.
       - When deploying code, these migration scripts are executed in order to keep the staging/production database schema in sync with the application code.

================================================================================
                           ANGULAR MASTER NOTES (v2 to v20)
================================================================================

1. ANGULAR BASICS & ARCHITECTURE
--------------------------------------------------------------------------------
* What is Angular?
  Angular is an open-source, front-end web framework developed and maintained by 
  Google. Based on TypeScript, it is highly suited for enterprise-grade web 
  applications due to its modularity, reusability, and maintainability.

* Architecture Paradigm:
  Modern Angular (v2+) uses a Component-Based Architecture. It has moved away 
  from the traditional MVC (Model-View-Controller) / MVVM pattern used in 
  legacy AngularJS (v1.x).

* Powerful Core Features:
  - Component-based structure.
  - Built-in Dependency Injection (DI).
  - Declarative templates with modern control flow.
  - Native client-server communication via HttpClient.
  - End-to-end tooling (Angular CLI).
  - Built-in animation, form validation, and routing.
  - Standardized state management primitives (Signals).

* Modules (NgModule) vs. Standalone Components (Angular 15+):
  - NgModule (Traditional): A container to declare and group related components, 
    directives, pipes, and configure providers. Every app traditionally had at 
    least one root module (AppModule).
  - Standalone Components (Modern): Introduced as a developer preview in Angular 14
    and stable since Angular 15. Components can be declared with `standalone: true`.
    They import their dependencies directly in their `@Component` decorator, making
    NgModules entirely optional. Apps can now be bootstrapped directly using
    `bootstrapApplication(AppComponent)`.

* Bootstrapping a Standalone Application:
  ```typescript
  // main.ts
  import { bootstrapApplication } from '@angular/platform-browser';
  import { AppComponent } from './app/app.component';
  import { provideRouter } from '@angular/router';
  import { routes } from './app/app.routes';
  import { provideHttpClient } from '@angular/common/http';

  bootstrapApplication(AppComponent, {
    providers: [
      provideRouter(routes),
      provideHttpClient()
    ]
  }).catch(err => console.error(err));
  ```


2. DIRECTIVES & PIPES
--------------------------------------------------------------------------------
* Directives:
  Directives are classes that add custom behavior to elements in the DOM.
  There are three main types of directives in Angular:

  1. Component Directives: Directives with a template. They are declared using 
     the `@Component` decorator (which inherits from `@Directive`).
  2. Structural Directives: Manipulate the layout of the DOM by adding, removing, 
     or replacing elements. Identified by a leading asterisk (`*`), such as 
     `*ngIf` and `*ngFor` (or legacy custom structural directives).
  3. Attribute Directives: Modify the appearance or behavior of an existing DOM 
     element (e.g. `ngStyle`, `ngClass`).

* Structural Directives Code Examples:
  - `*ngIf`:
    ```html
    <div *ngIf="isReady" class="display_name">  
      {{name}}  
    </div>  
    ```
  - `*ngFor`:
    ```html
    <div class="details" *ngFor="let x of details" >  
      <p>{{x.name}}</p>  
      <p>{{x.address}}</p>  
      <p>{{x.age}}</p>  
    </div>  
    ```

* Custom Attribute Directive Example:
  You can generate a directive via the CLI: `ng g directive yellowBackground`
  
  ```typescript
  // yellow-background.directive.ts
  import { Directive, ElementRef } from '@angular/core';
  
  @Directive({  
    selector: '[appYellowBackground]',
    standalone: true
  })  
  export class YellowBackgroundDirective {  
    constructor(el: ElementRef) {  
      el.nativeElement.style.backgroundColor = 'yellow';  
    }  
  }
  ```
  Usage in template:
  ```html
  <p appYellowBackground>Hello JavaTpoint</p>  
  ```

* What can Custom Directives be applied to?
  - Element directives: Directive activates when a matching element is encountered.
  - Attribute: Directive activates when a matching attribute is found on an element (most common).
  - CSS: Directive activates when a matching CSS style is encountered.
  - Comment: Directive activates when a matching HTML comment is encountered.

* Pipes (formerly Filters in AngularJS):
  Pipes are used to transform template data before rendering it.
  
  - Built-in Pipes:
    - `currency`: Formats numbers to currency. E.g., `{{ price | currency }}`
    - `date`: Formats date strings/objects. E.g., `{{ today | date:'medium' }}`
    - `json`: Formats an object into a JSON string. E.g., `<pre>{{ user | json }}</pre>`
    - `slice`: Selects a subset of an array or string. E.g., `{{ text | slice:0:10 }}`
    - `lowercase` / `uppercase`: Formats text casing.
    - `number`: Formats decimal/integer numbers.
  
  - Note on Filtering/Sorting: Modern Angular does not include built-in `filter` 
    or `orderBy` pipes because they are performance anti-patterns. They re-evaluate 
    on every change detection cycle, slowing down the UI. Logic for sorting and 
    filtering lists should be handled in the component class instead.

* Pure vs. Impure Pipes:
  - Pure Pipes (Default, `{ pure: true }`): Run only when the input reference changes.
    ```typescript
    @Pipe({ name: 'uppercasePipe', pure: true })
    export class UppercasePipe implements PipeTransform {
      transform(value: string): string {
        return value.toUpperCase();
      }
    }
    ```
  - Impure Pipes (`{ pure: false }`): Run on every change detection cycle (can be 
    expensive).


3. DATA BINDING & COMPONENT COMMUNICATION
--------------------------------------------------------------------------------
* Data Binding Types:
  - String Interpolation (`{{ value }}`): One-way from Component to View.
  - Property Binding (`[property]="value"`): One-way from Component to DOM property.
  - Event Binding (`(event)="handler()"`): One-way from View to Component.
  - Two-Way Data Binding (`[(ngModel)]="value"`): Bidirectional synchronization. 
    Requires `FormsModule`.

* Sharing Data Between Components:
  1. Parent to Child (Inputs):
     - Legacy: `@Input() title: string = '';`
     - Modern (Signal-based): `title = input<string>('');` (Read-only signal)
     - Usage: `<app-card [title]="'User Info'"></app-card>`

  2. Child to Parent (Outputs):
     - Legacy: `@Output() clicked = new EventEmitter<string>();`
     - Modern: `clicked = output<string>();`
     - Usage: `<app-card (clicked)="handleClick($event)"></app-card>`

  3. Custom Two-Way Binding (Inputs + Outputs):
     - Naming convention: An input property `status` paired with an output 
       `statusChange` lets the parent bind via `[(status)]="isActive"`.
     ```typescript
     @Input() status: boolean = false;
     @Output() statusChange = new EventEmitter<boolean>();
     ```
     - Usage: `<app-toggle [(status)]="isActive"></app-toggle>`

  4. Child to Parent via ViewChild:
     - Access child instances or elements directly:
     ```typescript
     @ViewChild('chart') chart!: ChartComponent;
     
     ngAfterViewInit() {
       this.chart.render();
     }
     ```

  5. Unrelated Components:
     - Sharing via a shared Service (using RxJS Subjects/BehaviorSubjects or Signals).
     - Global State Management libraries (e.g. NgRx).
     - Event Bus pattern (centralized event hub using an RxJS Subject).

* Content Projection (`ng-content`):
  Allows a parent to pass arbitrary HTML/components into a child's template.
  ```html
  <!-- child.component.html -->
  <div class="card">
    <ng-content select="[header]"></ng-content>  <!-- Named slot -->
    <ng-content></ng-content>                    <!-- Default slot -->
  </div>
  ```
  ```html
  <!-- Parent usage: -->
  <app-card>
    <h2 header>My Title</h2>   <!-- Goes into named slot -->
    <p>Body content</p>        <!-- Goes into default slot -->
  </app-card>
  ```
  - Use `@ContentChild` / `@ContentChildren` (or signal `contentChild()`) to query
    projected nodes from the child component.
  - `ngProjectAs`: Lets a wrapper element masquerade as a different selector for
    slot matching.


4. LIFECYCLE HOOKS
--------------------------------------------------------------------------------
* When a component is created, Angular runs it through a series of lifecycle phases:

  1. `ngOnChanges()`: Called before `ngOnInit()` and whenever `@Input()` references 
     change. Receives a `SimpleChanges` object mapping changed properties.
     ```typescript
     @Input() user!: User;
     
     ngOnChanges(changes: SimpleChanges) {
       if (changes['user']) {
         this.updateProfileForm(changes['user'].currentValue);
       }
     }
     ```
  2. `ngOnInit()`: Called once after the first `ngOnChanges()`. Ideal for 
     initialization logic and loading initial API data.
     ```typescript
     ngOnInit() {
       this.loadData();
     }
     ```
  3. `ngDoCheck()`: Called during every change detection cycle. Used to detect 
     changes that Angular's default algorithm misses.
     ```typescript
     ngDoCheck() {
       if (this.settings.theme !== this.prevTheme) {
         this.applyTheme(this.settings.theme);
         this.prevTheme = this.settings.theme;
       }
     }
     ```
  4. `ngAfterContentInit()`: Called once after projected content (`<ng-content>`) 
     is projected into the component.
     ```typescript
     @ContentChild('title') titleEl!: ElementRef;
     
     ngAfterContentInit() {
       console.log('Title text on first load:', this.titleEl.nativeElement.textContent);
     }
     ```
  5. `ngAfterContentChecked()`: Called after `ngAfterContentInit` and every 
     subsequent `ngDoCheck()`. Responds after projected content is checked.
     ```typescript
     ngAfterContentChecked() {
       console.log('Projected content re-checked');
     }
     ```
  6. `ngAfterViewInit()`: Called once after the component's view and child 
     views are fully initialized. Crucial for direct DOM queries.
     ```typescript
     @ViewChild('chart') chart!: ChartComponent;
     
     ngAfterViewInit() {
       this.chart.render();
     }
     ```
  7. `ngAfterViewChecked()`: Called after `ngAfterViewInit` and subsequent 
     `ngDoCheck()` cycles. Responds after child views are checked.
  8. `ngOnDestroy()`: Called just before Angular destroys the component. Used to 
     clean up subscriptions, timers, and prevent memory leaks.
     ```typescript
     ngOnDestroy() {
       this.resizeSub?.unsubscribe();
       clearInterval(this.autoSaveId);
     }
     ```

* Execution Order:
  `ngOnChanges` → `ngOnInit` → `ngDoCheck` → `ngAfterContentInit` → 
  `ngAfterContentChecked` → `ngAfterViewInit` → `ngAfterViewChecked` → `ngOnDestroy`

* Constructor vs. ngOnInit:
  - Constructor: A native TypeScript class feature. Only use it for Dependency Injection.
  - ngOnInit: An Angular-specific hook. Use it for data initialization and binding logic, 
    as inputs are not yet available in the constructor.


5. CHANGE DETECTION & PERFORMANCE OPTIMIZATION
--------------------------------------------------------------------------------
* Change Detection Strategies:
  1. Default Strategy: Angular checks every component in the tree from top to bottom 
     whenever an asynchronous event (DOM event, HTTP response, timer) occurs.
  2. OnPush Strategy (`ChangeDetectionStrategy.OnPush`): Angular only checks the 
     component and its subtree when:
     - An `@Input()` reference changes.
     - An event originates from the component or its children.
     - An Observable bound with the `async` pipe emits a new value.
     - `markForCheck()` is called manually on `ChangeDetectorRef`.

* Enabling OnPush:
  ```typescript
  @Component({
    selector: 'app-user-card',
    templateUrl: './user-card.component.html',
    changeDetection: ChangeDetectionStrategy.OnPush
  })
  export class UserCardComponent {}
  ```

* markForCheck() vs. detectChanges():
  - `markForCheck()`: Marks the component and its ancestors as needing change 
    detection. It will run during the next normal cycle.
    ```typescript
    constructor(private cd: ChangeDetectorRef) {}
    someMethod() {
      this.data.prop = 'new value'; // Mutating property without reference change
      this.cd.markForCheck();
    }
    ```
  - `detectChanges()`: Synchronously forces change detection on the current component 
    and its children immediately. Use with caution.
    ```typescript
    constructor(private cd: ChangeDetectorRef) {}
    updateView() {
      this.data = newData;
      this.cd.detectChanges();
    }
    ```
  - `detach()` and `reattach()`: Can be used to disable and resume change detection 
    manually for custom performance tuning.

* Immutable Objects and OnPush:
  OnPush relies heavily on object reference changes. Do not mutate existing objects.
  ```typescript
  // ❌ BAD: No reference change (Will not trigger OnPush CD)
  this.parentData.message = 'mutated';

  // ✅ GOOD: Reference change (Will trigger OnPush CD)
  this.parentData = { message: 'new message object' };
  ```

* Performance Optimization Checklist:
  1. trackBy in *ngFor: Avoids full DOM re-renders when list data changes.
     ```html
     <li *ngFor="let item of items; trackBy: trackById">{{ item.name }}</li>
     ```
     ```typescript
     trackById(index: number, item: any) {
       return item.id;
     }
     ```
  2. Avoid Logic in Templates: Inline template calculations execute on every single 
     change detection cycle.
     ```html
     <!-- ❌ BAD -->
     {{ users.filter(u => u.isActive).length }}
     
     <!-- ✅ GOOD -->
     {{ activeUserCount }}
     ```
  3. Use ng-container to Avoid Extra DOM Elements:
     ```html
     <ng-container *ngIf="isLoggedIn">
       <button>Logout</button>
     </ng-container>
     ```
  4. Prefer One-Way Binding: Avoid unnecessary double-binding if a one-way binding 
     is enough.
     ```html
     <!-- ❌ BAD: Triggers change detection twice -->
     <input [(ngModel)]="name" />
     
     <!-- ✅ GOOD -->
     <input [value]="name" (input)="name = $event.target.value" />
     ```


6. ROUTING & NAVIGATION
--------------------------------------------------------------------------------
* Routing Setup:
  ```typescript
  const routes: Routes = [
    { path: '', redirectTo: 'home', pathMatch: 'full' },
    { path: 'home', component: HomeComponent },
    { path: '**', component: NotFoundComponent } // Wildcard route for 404
  ];
  ```

* Template Navigation:
  - Using Directives: `<a routerLink="/login" routerLinkActive="active">Login</a>`
  - Programmatic Navigation:
    ```typescript
    constructor(private router: Router) {}
    goToDashboard() {
      this.router.navigate(['/dashboard']);
    }
    ```

* Route Parameters & Query Params:
  - Dynamic parameters: `{ path: 'product/:id', component: ProductComponent }`
    ```typescript
    constructor(private route: ActivatedRoute) {}
    ngOnInit() {
      const id = this.route.snapshot.paramMap.get('id');
    }
    ```
  - Query Parameters:
    ```typescript
    // Navigating: /products?page=2&sort=price
    this.router.navigate(['/products'], { queryParams: { page: 2, sort: 'price' } });
    
    // Reading:
    this.route.queryParamMap.subscribe(params => {
      const page = params.get('page');
    });
    ```

* Route Guards (Modern Functional Guards):
  - `CanActivateFn`: Prevents entering a route.
    ```typescript
    export const authGuard: CanActivateFn = () => {
      const router = inject(Router);
      const loggedIn = localStorage.getItem('loggedIn') === 'true';
      return loggedIn ? true : router.createUrlTree(['/login']);
    };
    ```
  - `CanDeactivateFn`: Prevents leaving a route (e.g. check unsaved changes).
    ```typescript
    export const canExitForm: CanDeactivateFn<EditComponent> = (component) => {
      return component.hasUnsavedChanges ? confirm('Discard changes?') : true;
    };
    ```
  - `CanMatchFn` (Modern replacement for `CanLoadFn`):
    ```typescript
    export const adminGuard: CanMatchFn = () => {
      return localStorage.getItem('role') === 'admin';
    };
    ```
  - `ResolveFn`: Pre-fetches data before component activates.
    ```typescript
    export const userResolver: ResolveFn<User> = () => {
      return inject(UserService).getUserProfile();
    };
    ```
    Accessing resolved data in the component:
    ```typescript
    this.route.data.subscribe(data => {
      console.log(data['user']);
    });
    ```
  - `CanActivateChild`: Applies check to all child routes.
    ```typescript
    {
      path: 'admin',
      canActivateChild: [authGuard],
      children: [
        { path: 'users', component: UsersComponent }
      ]
    }
    ```


7. DEPENDENCY INJECTION (DI)
--------------------------------------------------------------------------------
* DI Pattern:
  Angular uses dependency injection to supply components, directives, and services 
  with the dependencies they need. Services are registered with providers (typically 
  instantiated once as singletons during the lifetime of the application).

* Injection Methods:
  1. Constructor Injection (Traditional):
     ```typescript
     constructor(private http: HttpClient) {}
     ```
  2. `inject()` Function (Modern - Angular 14+):
     Allows injection directly in class property declarations, functional guards, 
     and custom functions:
     ```typescript
     http = inject(HttpClient);
     ```


8. HTTP CLIENT & ERROR HANDLING
--------------------------------------------------------------------------------
* HttpClient:
  The reactive HTTP client from `@angular/common/http` that wraps request APIs and 
  returns RxJS Observables.

* Best Practices for Error Handling:
  1. Local Error Handling: Pass an observer object to `.subscribe()` containing 
     `next` and `error` handlers:
     ```typescript
     this.userService.getProfile().subscribe({
       next: (data) => this.profile = data,
       error: (err) => this.error = err
     });
     ```
  2. Global Error Handling: Create a custom class that implements Angular's `ErrorHandler` 
     to capture uncaught runtime exceptions globally.
  3. Http Interceptors: Intercept requests/responses globally (e.g. to attach auth tokens
     or catch specific status code errors like 401/403).
     - Modern Functional Interceptor (v15+):
       ```typescript
       export const authInterceptor: HttpInterceptorFn = (req, next) => {
         const token = inject(AuthService).getToken();
         const authReq = req.clone({
           headers: req.headers.set('Authorization', `Bearer ${token}`)
         });
         return next(authReq);
       };

       // Registration in main.ts:
       provideHttpClient(withInterceptors([authInterceptor]))
       ```
     - For error handling in an interceptor:
       ```typescript
       return next(req).pipe(
         catchError(err => {
           if (err.status === 401) inject(Router).navigate(['/login']);
           return throwError(() => err);
         })
       );
       ```


9. FORMS IN ANGULAR
--------------------------------------------------------------------------------
* Reactive Forms:
  Model-driven forms where the form structure is built in TypeScript and mapped to the template.

* Basic Setup Example:
  ```typescript
  // Using FormBuilder shorthand:
  this.form = this.fb.group({
    name: ['', [Validators.required, Validators.minLength(3)]],
    email: ['', [Validators.required, Validators.email]]
  });
  ```

* setValue vs. patchValue:
  - `setValue()`: Updates values of all form controls. Requires full form structure.
    ```typescript
    this.form.setValue({ name: 'Sahil', email: 'sahil@example.com' });
    ```
  - `patchValue()`: Updates only provided fields.
    ```typescript
    this.form.patchValue({ name: 'Sahil' });
    ```

* Custom Validators:
  - Sync Custom Validator:
    ```typescript
    function noSpecialChars(control: AbstractControl): ValidationErrors | null {
      return /^[a-zA-Z0-9]*$/.test(control.value) ? null : { specialChars: true };
    }
    ```
  - Async Custom Validator:
    ```typescript
    function checkUsername(control: AbstractControl): Observable<ValidationErrors | null> {
      return api.checkUser(control.value).pipe(
        map(exists => exists ? { usernameTaken: true } : null)
      );
    }
    ```
  - Form-Level Validator (e.g. comparing two fields):
    ```typescript
    this.form = this.fb.group({
      password: [''],
      confirmPassword: ['']
    }, { validators: passwordMatchValidator });
    ```

* FormArray (Dynamic Lists):
  Used to handle dynamically sized lists of controls.
  ```typescript
  skills: this.fb.array([this.fb.control('')]);
  
  addSkill() {
    this.skills.push(this.fb.control('Angular'));
  }
  ```

* valueChanges vs. statusChanges:
  - `valueChanges`: Emits whenever a form value changes.
  - `statusChanges`: Emits the validation status (VALID, INVALID, PENDING, DISABLED).

* Enable/Disable Form Fields:
  ```typescript
  this.form.get('email')?.disable();
  this.form.get('email')?.enable();
  ```
  Note: Disabled controls are excluded from `form.value`. To fetch all fields, 
  including disabled ones, use: `this.form.getRawValue()`.

* Validators.compose():
  Used to combine multiple validator functions conditionally or dynamically:
  ```typescript
  Validators.compose([Validators.required, Validators.minLength(3)])
  ```

* Programmatic Validation Triggering:
  ```typescript
  this.form.markAllAsTouched(); // Triggers UI validation errors
  ```


10. RxJS & REACTIVE PROGRAMMING
--------------------------------------------------------------------------------
* Promises vs. Observables:
  - Promise: Emits a single value, execution starts immediately (not lazy), 
    and cannot be cancelled natively.
  - Observable: Emits multiple values over time, lazy (does not execute until 
    subscribed), and can be cancelled by unsubscribing.

* Debouncing & Throttling in Angular using RxJS:
  ```typescript
  this.form = this.fb.group({
    debounceSearch: [null],
    throttleSearch : [null]
  });

  this.form.get('debounceSearch')!.valueChanges
    .pipe(debounceTime(500))
    .subscribe(value => console.log('Debounced:', value));

  this.form.get('throttleSearch')!.valueChanges
    .pipe(throttleTime(500))
    .subscribe(value => console.log('Throttled:', value));
  ```

* Subjects (Multicast Observables):
  Subjects act as both an Observable and an Observer:
  - `Subject`: Standard multicast stream. Late subscribers miss previously emitted values.
  - `BehaviorSubject`: Requires an initial value and replays the last emitted value 
    to new subscribers immediately.
  - `ReplaySubject`: Buffers a history of values and replays them to new subscribers.
  - `AsyncSubject`: Emits only the final value of the stream upon completion.

* Unsubscribe Strategies:
  1. Manual unsubscription: `this.sub.unsubscribe();`
  2. `takeUntil()` with Subject:
     ```typescript
     private destroy$ = new Subject<void>();
     ngOnInit() {
       this.api.getData().pipe(takeUntil(this.destroy$)).subscribe();
     }
     ngOnDestroy() {
       this.destroy$.next();
       this.destroy$.complete();
     }
     ```
  3. `takeUntilDestroyed()` (v16+):
     ```typescript
     constructor() {
       this.api.getData().pipe(takeUntilDestroyed()).subscribe();
     }
     ```
  4. `AsyncPipe` in templates: Handles subscription and unsubscription automatically.


11. CODE SPLITTING & LAZY LOADING
--------------------------------------------------------------------------------
* Code Splitting:
  Dividing JavaScript code into smaller chunks (modules/components) so that the 
  browser only downloads what is needed for the active route.

* Lazy Loading:
  Loads features on demand rather than eagerly on application startup.
  - Lazy Loading NgModules:
    ```typescript
    { path: 'admin', loadChildren: () => import('./admin/admin.module').then(m => m.AdminModule) }
    ```
  - Lazy Loading Standalone Components (v14+):
    ```typescript
    { path: 'dashboard', loadComponent: () => import('./dashboard/dashboard.component').then(m => m.DashboardComponent) }
    ```

* Dynamic Lazy Loading (Outside of Routing):
  You can load components on the fly (e.g. for modals or dialogs) inside your TypeScript logic:
  ```typescript
  @ViewChild('container', { read: ViewContainerRef }) vcr!: ViewContainerRef;

  async loadModal() {
    const { LazyModalComponent } = await import('./modal/lazy-modal.component');
    this.vcr.clear();
    const compRef = this.vcr.createComponent(LazyModalComponent);
    compRef.instance.id = 1; // Pass inputs manually
  }
  ```

* Selective Preloading Strategy:
  Downloads lazy-loaded code in the background after the initial app is stable.
  
  1. Define metadata in route configuration:
     ```typescript
     const routes: Routes = [
       {
         path: 'dashboard',
         loadChildren: () => import('./dashboard/dashboard.module').then(m => m.DashboardModule),
         data: { preload: true } // should preload
       }
     ];
     ```
  2. Create the custom strategy service:
     ```typescript
     import { Injectable } from '@angular/core';
     import { Route, PreloadingStrategy } from '@angular/router';
     import { Observable, of } from 'rxjs';

     @Injectable({ providedIn: 'root' })
     export class SelectivePreloadingStrategy implements PreloadingStrategy {
       preload(route: Route, load: () => Observable<any>): Observable<any> {
         return route.data?.['preload'] ? load() : of(null);
       }
     }
     ```
  3. Register the strategy in AppRoutingModule:
     ```typescript
     @NgModule({
       imports: [
         RouterModule.forRoot(routes, {
           preloadingStrategy: SelectivePreloadingStrategy
         })
       ]
     })
     export class AppRoutingModule {}
     ```


12. MODERN ANGULAR (v16 - v20) ADDITIONS
--------------------------------------------------------------------------------
* Angular Signals:
  A fine-grained reactivity system introduced in v16. Updates only the specific DOM 
  nodes that depend on the changed signal, bypassing global Zone.js checks.
  
  ```typescript
  count = signal(0);
  doubleCount = computed(() => this.count() * 2);

  increment() {
    this.count.update(c => c + 1);
  }
  ```

* `effect()`:
  Runs a side-effect function whenever its dependent signals change. Must be called
  in an injection context (constructor or `runInInjectionContext`).
  ```typescript
  constructor() {
    effect(() => {
      console.log('Count is now:', this.count()); // Re-runs whenever count changes
    });
  }
  ```
  - Use `allowSignalWrites: true` option if you need to write to signals inside an effect.
  - Prefer `computed()` over `effect()` for deriving state; use `effect()` only for
    true side effects (logging, DOM manipulation, syncing to localStorage).

* Signal Inputs, Model, & Outputs:
  - Read-only Signal Inputs (v17.1+):
    ```typescript
    username = input<string>('Guest'); // Read-only Signal<string>
    userId = input.required<string>(); // Required input signal
    ```
  - Model Inputs (v17.2+): Support signal-based two-way data binding.
    ```typescript
    volume = model<number>(50); // Bind in parent using [(volume)]="currentVolume"
    ```
  - Signal Outputs (v17.3+):
    ```typescript
    clicked = output<string>();
    this.clicked.emit('data');
    ```

* RxJS ↔ Signal Interoperability (`@angular/core/rxjs-interop`):
  ```typescript
  import { toSignal, toObservable } from '@angular/core/rxjs-interop';

  // Observable → Signal (subscribe managed automatically):
  data = toSignal(this.http.get<User[]>('/api/users'), { initialValue: [] });

  // Signal → Observable (emits whenever signal changes):
  count$ = toObservable(this.count);
  ```
  - `toSignal()` must be called in an injection context.
  - Provide `initialValue` or `requireSync: true` to avoid the `undefined` initial state.

* Signal Queries (v17.2+):
  Modern alternatives to ViewChild/ContentChild.
  ```typescript
  box = viewChild<ElementRef>('boxEl'); // returns Signal<ElementRef | undefined>
  boxes = viewChildren('boxEl');       // returns Signal<readonly ElementRef[]>
  ```

* Block-Based Control Flow (v17+):
  Provides structured HTML controls without using directive imports:
  
  ```html
  @if (isLoggedIn) {
    <p>Welcome!</p>
  } @else {
    <button>Log In</button>
  }

  @for (user of users; track user.id) {
    <div>{{ user.name }}</div>
  } @empty {
    <p>No users found.</p>
  }

  @switch (status) {
    @case ('active')   { <span class="green">Active</span> }
    @case ('inactive') { <span class="red">Inactive</span> }
    @default           { <span>Unknown</span> }
  }
  ```

* Deferred Views (`@defer` - v17+):
  Lazy-loads sections of a template automatically based on trigger conditions.
  Supported trigger types:
  | Trigger              | Description                                      |
  |----------------------|--------------------------------------------------|
  | `on idle`            | When the browser is idle                         |
  | `on viewport`        | When the block enters the viewport               |
  | `on interaction`     | On first user click or keydown on the block      |
  | `on hover`           | On mouseover                                     |
  | `on timer(2s)`       | After a fixed delay                              |
  | `when condition`     | When a boolean expression becomes true           |
  | `prefetch on idle`   | Pre-fetches assets while the block is deferred   |

  ```html
  @defer (on viewport; prefetch on idle) {
    <app-heavy-chart />
  } @placeholder {
    <div>Loading chart...</div>
  } @loading (minimum 200ms) {
    <app-spinner />
  } @error {
    <p>Failed to load chart.</p>
  }
  ```

* Zoneless Change Detection (v18+):
  Reduces bundle size and improves performance by running without `Zone.js`.
  Enable by adding `provideExperimentalZonelessChangeDetection()` inside the
  `bootstrapApplication` config in `main.ts`. Became stable in Angular v20.

* `linkedSignal()` (v19+):
  Creates a writable signal that automatically resets to a computed value when its
  source changes. Useful for dependent state (e.g. resetting a selection when a list changes).
  ```typescript
  items = signal(['A', 'B', 'C']);
  selectedItem = linkedSignal(() => this.items()[0]); // Resets when items changes

  selectItem(item: string) {
    this.selectedItem.set(item); // Can also be set manually
  }
  ```

* `resource()` / `rxResource()` (v19+):
  Built-in primitives for async data loading directly tied to signals.
  Exposes `value`, `status`, `error`, and `isLoading` as signals.
  ```typescript
  // resource() — uses a plain Promise loader:
  userResource = resource({
    request: () => ({ id: this.userId() }),
    loader: ({ request }) => fetch(`/api/users/${request.id}`).then(r => r.json())
  });

  // rxResource() — uses an Observable loader:
  userResource = rxResource({
    request: () => ({ id: this.userId() }),
    loader: ({ request }) => this.http.get<User>(`/api/users/${request.id}`)
  });

  // Template usage:
  // @if (userResource.isLoading()) { <spinner /> }
  // @if (userResource.value()) { {{ userResource.value().name }} }
  ```


13. DIRECTIVES — ADVANCED FEATURES
--------------------------------------------------------------------------------
* `HostListener` & `HostBinding`:
  Used inside a directive or component to listen to host element events or bind
  host element properties.
  ```typescript
  @Directive({ selector: '[appHighlight]', standalone: true })
  export class HighlightDirective {
    @HostBinding('style.backgroundColor') bgColor = 'transparent';

    @HostListener('mouseenter') onEnter() { this.bgColor = 'yellow'; }
    @HostListener('mouseleave') onLeave() { this.bgColor = 'transparent'; }
  }
  ```
  Modern equivalent — use `host` metadata in the decorator:
  ```typescript
  @Directive({
    selector: '[appHighlight]',
    standalone: true,
    host: {
      '(mouseenter)': 'onEnter()',
      '(mouseleave)': 'onLeave()',
      '[style.backgroundColor]': 'bgColor'
    }
  })
  ```

* `ViewEncapsulation`:
  Controls how component styles are scoped to the component's DOM.
  | Mode         | Description                                                      |
  |--------------|------------------------------------------------------------------|
  | `Emulated`   | Default. Angular adds unique attributes to scope styles          |
  | `None`       | Styles are global — can affect other components                  |
  | `ShadowDom`  | Uses native Shadow DOM; true CSS isolation                       |
  ```typescript
  @Component({
    selector: 'app-card',
    encapsulation: ViewEncapsulation.None // Styles leak globally
  })
  ```

* `DestroyRef`:
  A modern, injectable token (v16+) that lets you register cleanup callbacks
  without implementing `ngOnDestroy`. Works in any injection context.
  ```typescript
  constructor(private destroyRef: DestroyRef) {
    destroyRef.onDestroy(() => {
      this.subscription.unsubscribe();
      console.log('Cleaned up!');
    });
  }
  ```
  - Preferred over `ngOnDestroy` for services and functional code, as it keeps
    cleanup logic co-located with setup logic.

* `input()` with `transform` (v17.1+):
  Transforms an incoming input value before it is stored in the signal.
  ```typescript
  import { input, numberAttribute, booleanAttribute } from '@angular/core';

  // Auto-coerces string attribute "42" → number 42 from HTML:
  count = input(0, { transform: numberAttribute });

  // Auto-coerces string attribute "true" → boolean:
  disabled = input(false, { transform: booleanAttribute });

  // Custom transform:
  label = input('', { transform: (v: string) => v.trim().toUpperCase() });
  ```

================================================================================
                            BROWSER APIs & RUNTIME NOTES
================================================================================

1. DOM, BOM, & DOM NAVIGATION
--------------------------------------------------------------------------------
* DOM (Document Object Model):
  - Represents all page content as objects that can be dynamically modified.
  - The `document` object is the main entry point to the page. We can change or create 
    anything on the page using it.
    ```javascript
    // Example: Change the background color to red
    document.body.style.background = "red";
    ```

* BOM (Browser Object Model):
  - Represents additional objects provided by the browser (host environment) for working 
    with everything except the document itself.
  - Examples:
    - `navigator`: Provides background information about the browser and operating system.
      - `navigator.userAgent`: Information about the current browser agent.
      - `navigator.platform`: Information about the OS platform (Windows, Mac, Linux).
    - `location`: Allows reading the current URL and redirecting the browser.

* DOM Tree Navigation:
  - According to the DOM, every HTML tag is an object. Nested tags are "children" of the enclosing one.
  - `document.body` is the object representing the `<body>` tag.
  - **Child nodes (or children)**: Direct nested elements (e.g. `<head>` and `<body>` are children of `<html>`).
    - Accessed via: `childNodes`, `firstChild`, `lastChild`.
  - **Descendants**: All elements nested inside a given one (children, their children, and so on recursively).

* Selecting Elements:
  - `document.getElementById(id)`: Returns the element matching the unique ID, regardless of where it is.
  - `elem.querySelectorAll(css)`: Returns all elements inside `elem` matching the given CSS selector.
    ```html
    <script>
      let elements = document.querySelectorAll('ul > li:last-child');
      for (let elem of elements) {
        alert(elem.innerHTML); // e.g. "test", "passed"
      }
    </script>
    ```


2. HTML ATTRIBUTES vs. DOM PROPERTIES
--------------------------------------------------------------------------------
- **Attributes**: What is written directly in the raw HTML markup.
- **Properties**: What is stored inside the instantiated DOM object.
- Standard attributes for one element can be unknown for another. Standard attributes are 
  mapped to DOM properties, while non-standard attributes are ignored.
  ```html
  <body id="body" type="...">
    <input id="input" type="text">
    <script>
      alert(input.type); // "text" (standard for input)
      alert(body.type);  // undefined (non-standard for body, DOM property not created)
    </script>
  </body>
  ```


3. BROWSER EVENTS & PROPAGATION
--------------------------------------------------------------------------------
* Browser Events:
  An event is a signal that something has occurred. All DOM nodes generate signals.
  - Mouse events: `click` (clicks/taps), `contextmenu` (right-clicks), `mouseover`/`mouseout` (hovering), 
    `mousedown`/`mouseup` (clicks in progress), `mousemove`.
  - Keyboard events: `keydown`, `keyup`.
  - Form events: `submit`, `focus`.
  - Document events: `DOMContentLoaded` (DOM fully built).
  - CSS events: `transitionend`.

* Event Bubbling:
  When an event happens on an element, it first runs the handlers on it, then on its parent, 
  and so on all the way up through its ancestors.

* Event Target (`event.target`):
  A handler on a parent element can always access where the event actually occurred.
  - `event.target`: The most deeply nested target element that initiated the event.
  - `this` (or `event.currentTarget`): The element currently executing the handler.

* Stopping Propagation:
  By default, events bubble up to `<html>`, `document`, and sometimes the `window`.
  - `event.stopPropagation()`: Stops the event bubble from moving further up the hierarchy.


4. WINDOW POPUPS & CROSS-WINDOW COMMUNICATION
--------------------------------------------------------------------------------
* Window Popups (`window.open`):
  - Syntax: `window.open(url, name, params)`
  - Opens a new window or tab. If sizes are provided in params, it opens a popup window.
  - Closing a popup: Call `win.close()`. Check if closed using `win.closed` (returns boolean).

* Cross-Window Communication & Same-Origin Policy:
  - Same-Origin Policy: Restricts windows and frames from accessing each other's data 
    unless they share the same origin (protocol, domain, and port) to prevent security/information theft.

* Cross-Window Messaging (`postMessage`):
  An interface that allows windows with different origins to communicate safely by exchanging data.
  - **Sending**: The sending window calls `.postMessage(data, targetOrigin)` on the receiving window.
    ```javascript
    win.postMessage("hello data", "https://gmail.com");
    ```
  - **Receiving**: The target window listens for the `message` event.
    ```javascript
    window.addEventListener("message", (event) => {
      if (event.origin !== "https://john-smith.com") return; // Verify origin
      console.log(event.data);
    });
    ```


5. BINARY DATA & FILE APIs
--------------------------------------------------------------------------------
* TextDecoder and TextEncoder:
  - `TextDecoder`: Decodes binary buffers into actual JavaScript strings.
    ```javascript
    let decoder = new TextDecoder("utf-8");
    let uint8Array = new Uint8Array([72, 101, 108, 108, 111]);
    alert(decoder.decode(uint8Array)); // "Hello"
    ```
  - `TextEncoder`: Converts a string into a Uint8Array of bytes.
    ```javascript
    let encoder = new TextEncoder();
    let uint8Array = encoder.encode("Hello"); // [72, 101, 108, 108, 111]
    ```

* ArrayBuffer:
  - A reference to a fixed-length, contiguous raw memory area. It does not represent data values 
    directly; we must create a "view" object (like `Uint8Array`, `DataView`) to read/write to it.
    ```javascript
    let buffer = new ArrayBuffer(16);
    alert(buffer.byteLength); // 16
    ```

* Blob (Binary Large Object):
  - Represents raw binary data *with a MIME type* (unlike ArrayBuffers which are raw arrays).
  - Convenient for uploads and downloads.
  - Conversion:
    - Create a Blob: `new Blob([typedArray], { type: 'text/plain' })`
    - Convert to ArrayBuffer: `blob.arrayBuffer()`

* Blob as URL:
  - Can be mapped to a local Object URL for elements like `<a>` or `<img>`.
    ```html
    <a download="hello.txt" href="#" id="link">Download</a>
    <script>
      let blob = new Blob(["Hello, world!"], { type: 'text/plain' });
      link.href = URL.createObjectURL(blob);
    </script>
    ```

* Blob to Base64 (Data URLs):
  - Converts a Blob into a base64 encoded string using readable ASCII characters. Used in inline data-URLs.

* File & FileReader:
  - `File`: Inherits from `Blob`, extended with filesystem properties (`name`, `lastModified`). Obtained via 
    `new File(...)` constructor or user selection in `<input type="file">`.
  - `FileReader`: An event-driven helper object used to read data from Blobs/Files in three formats:
    1. String: `readAsText(blob)`
    2. ArrayBuffer: `readAsArrayBuffer(blob)`
    3. Data URL (Base64): `readAsDataURL(blob)`


6. THE FETCH API
--------------------------------------------------------------------------------
* Fetch Syntax:
  - Enables asynchronous network requests ("AJAX").
  - Syntax: `let promise = fetch(url, [options])`
    ```javascript
    let response = await fetch(url, options); // Resolves with response headers
    let result = await response.json();      // Reads body as JSON
    
    // Promise format:
    fetch(url, options)
      .then(response => response.json())
      .then(result => console.log(result));
    ```

* Response Headers:
  - Accessed via the `response.headers` object.
    ```javascript
    let response = await fetch(url);
    console.log(response.headers.get('Content-Type'));
    ```

* Request Headers:
  - Configured in the fetch options:
    ```javascript
    fetch(url, {
      headers: { Authorization: 'Bearer token' }
    });
    ```

* FormData:
  - Used to capture HTML forms or programmatically build multi-part key-value payloads to send via fetch.
    ```javascript
    let formData = new FormData();
    formData.append('username', 'Sahil');
    formData.append('avatar', fileBlob, 'avatar.png'); // 3-arg syntax for files
    formData.set('role', 'admin'); // Overwrites existing keys
    ```
  - Form data is sent with the header `Content-Type: multipart/form-data`.

* Download Progress:
  - Tracked via `response.body` (a `ReadableStream`). Read chunks in a loop and accumulate lengths.
  - Note: Fetch cannot track *upload* progress. Use `XMLHttpRequest` for upload tracking.

* Abort Controller:
  - Used to cancel ongoing fetch requests.
    ```javascript
    let controller = new AbortController();
    setTimeout(() => controller.abort(), 1000); // Abort after 1s
    
    try {
      let response = await fetch(url, { signal: controller.signal });
    } catch(err) {
      if (err.name === 'AbortError') {
        alert("Request aborted!");
      }
    }
    ```


7. CROSS-ORIGIN RESOURCE SHARING (CORS)
--------------------------------------------------------------------------------
* Definition:
  CORS is a browser security mechanism that restricts cross-origin HTTP requests.

* "Safe" Requests (No Preflight):
  - Methods: `GET`, `POST`, or `HEAD`.
  - Headers: Only standard simple headers (like `Accept`, `Accept-Language`, `Content-Language`).
  - `Content-Type`: Only `application/x-www-form-urlencoded`, `multipart/form-data`, or `text/plain`.
  - Flow: Sent immediately. Browser includes the `Origin` header. Server replies with 
    `Access-Control-Allow-Origin` (matching the origin or `*` for anonymous requests) and 
    `Access-Control-Allow-Credentials: true` if credentials are required.

* "Unsafe" Requests (Preflight Issued):
  - Triggers on custom methods (PUT/DELETE) or custom headers.
  - **Preflight (OPTIONS)**: Browser issues a preliminary request checking permissions.
    - Headers: `Access-Control-Request-Method`, `Access-Control-Request-Headers`.
    - Server must reply with `200 OK` and headers: `Access-Control-Allow-Methods`, 
      `Access-Control-Allow-Headers`, and `Access-Control-Max-Age` (caching permissions).
  - The actual request is then sent.


8. URL OBJECTS & XMLHTTPREQUEST
--------------------------------------------------------------------------------
* URL Object:
  - Convenient parser for path strings.
  - Syntax: `new URL(url, [base])`
  - Components: `href`, `protocol` (ends with `:`), `search` (starts with `?`), `hash` (starts with `#`).

* XMLHttpRequest (XHR):
  - Legacy built-in browser object to perform HTTP requests.
  - Primary modern use case: Tracking file upload progress (which fetch does not support) to implement 
    resumable file uploads by querying uploaded byte counts from servers.


9. POLLING, WEBSOCKETS, & SERVER-SENT EVENTS
--------------------------------------------------------------------------------
* Regular Polling:
  - Sending requests at fixed intervals (e.g. every 10s).
  - Inefficient: creates massive connection overhead and updates are delayed by the interval.

* Long Polling:
  - Persistent-like HTTP approach. Client requests; server holds connection open and only responds 
    when there is new data. Once responded, the client immediately opens a new request.
  - Good for rare updates, but suffers from request-header overhead on busy servers.

* WebSocket:
  - TCP-based protocol (`ws://`, `wss://`) providing bidirectional full-duplex communication.
  - **Rate Limiting (`bufferedAmount`)**:
    ```javascript
    // Check socket bufferedAmount before sending more data to prevent memory overflow
    setInterval(() => {
      if (socket.bufferedAmount == 0) {
        socket.send(moreData());
      }
    }, 100);
    ```

* Server-Sent Events (SSE):
  - Built-in `EventSource` class keeps a persistent, unidirectional text-only HTTP connection.
  - Offers automatic reconnect with retry timeout and `Last-Event-ID` tracking.


10. CLIENT-SIDE STORAGE
--------------------------------------------------------------------------------
* Cookies (`document.cookie`):
  - Small strings stored in the browser. Size limit is 4KB per cookie.
  - Set by servers via `Set-Cookie` headers, and automatically sent by browsers via `Cookie` headers.
  - Modification: `document.cookie = "user=John; path=/; domain=site.com; secure; samesite=strict";`
  - Attributes:
    - `path`: Path prefix constraint.
    - `domain`: Sets sub-domain visibility.
    - `expires` / `max-age`: Expiry timers. If absent, the cookie dies when the browser tab is closed.
    - `secure`: Transmit only over HTTPS.
    - `samesite`: Blocks cross-site request cookie transmission to mitigate CSRF (XSRF) attacks.

* Web Storage (LocalStorage & SessionStorage):
  - Keys and values must be strings. Not sent to server on requests, supporting larger capacities (~5MB).
  - Methods: `setItem(key, val)`, `getItem(key)`, `removeItem(key)`, `clear()`, `length`.
  - **LocalStorage**: Shared across tabs/windows of the same origin. Data never expires.
  - **SessionStorage**: Restricted to the current browser tab. Survives page refreshes but dies when the tab is closed.

* IndexedDB:
  - A highly powerful, asynchronous transactional database built into the browser.
  - Stores complex object data with key ranges, indexing, and supports transactions.
